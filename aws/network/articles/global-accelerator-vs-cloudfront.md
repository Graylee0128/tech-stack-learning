# Global Accelerator vs CloudFront：AWS 的兩條「快速通道」

> **核心比喻**：你要把東西從 A 地送到全球各地。CloudFront 是在各個城市設立倉庫（Cache），收到訂單馬上從最近的倉庫出貨；而 Global Accelerator 是幫你蓋一條「專屬磁浮列車」，把你的貨直接用最快的路線送到客戶手中。

---

## 第一層：本質區別

### OSI 層級的差異

| 特性 | Global Accelerator | CloudFront |
|------|---|---|
| **工作層級** | Layer 4（傳輸層） | Layer 7（應用層） |
| **核心職責** | **路由優化** | **內容分發** |
| **處理對象** | 任意 TCP/UDP 流量 | HTTP/HTTPS 內容 |
| **快取功能** | ❌ 不快取 | ✅ 主動快取 |
| **使用場景** | 網路路徑優化 | 內容邊緣分發 |

**通俗理解**：
- **GA**：像高速公路管制員，把車輛導到最快的路線
- **CF**：像便利店鏈，提前在各地備好常用商品

## 第二層：功能詳細對比

### 1. 協議支援（最核心的區別）

#### Global Accelerator：全協議支援 ✅
```
GA 支援的通訊方式：
├── HTTP/HTTPS ✅
├── TCP（任意應用）✅
├── UDP（實時應用）✅
│   ├── 遊戲（FPS、MOBA 等）
│   ├── 語音通話（VoIP）
│   ├── 影片會議
│   └── IoT（MQTT、AMQP）
└── 任何自定義協議 ✅

技術原理：
使用者 ─TCP 連線→ GA Edge 節點
                    ↓（AWS 高速骨幹網）
                  後端應用（TCP/UDP 轉發）
```

**為什麼 GA 能做到？**
- 在邊緣節點與用戶建立 TCP 連線
- 同時在邊緣節點與後端建立連線
- 中間用 AWS 私有骨幹網銜接
- 對應用層完全透明

#### CloudFront：HTTP/HTTPS 專科 ⚠️
```
CF 支援的協議：
├── HTTP ✅
├── HTTPS ✅
├── WebSocket ✅
│   └── 實時通訊（Chat、Live Updates）
├── TCP ❌（設計上不支援）
└── UDP ❌（完全不支援）
```

**為什麼 CF 不支援 TCP/UDP？**
```
CF 的設計初心：
內容分發網路（CDN）
  → 針對 Web 內容優化
  → 假設都是 HTTP/HTTPS
  → 邊緣快取的內容必須是可重複使用的
  → TCP/UDP 的實時流量無法快取
```

**考題驗證**
```
題目：應用使用 UDP 協議傳輸遊戲狀態，
     需要全球低延遲。應使用什麼服務？

A) CloudFront
B) Global Accelerator ✅
C) Route 53
D) ALB

答案：B - GA 唯一支援 UDP 的 AWS 加速服務
```

### 2. IP 定址策略

#### Global Accelerator：靜態 Anycast IP
```
核心特性：提供 2 個靜態 IP 地址
  - IP 1：主用
  - IP 2：備用（故障轉移）

全球路由行為：
┌─────────────┬─────────────┐
│ 倫敦用戶    │ 台北用戶    │
└──────┬──────┴──────┬──────┘
       ↓              ↓
    都連 1.2.3.4   都連 1.2.3.4 ← 同一個 IP！
       ↓              ↓
  倫敦邊緣節點    台北邊緣節點
       └──────────────┘
        （GA 自動路由，無感轉移）
```

**優勢**：
- ✅ 一個固定 IP，用戶端設定簡單
- ✅ 完美支援硬編碼 IP 的舊系統
- ✅ BYOIP：可使用企業自有 IP（企業遷移救星）

#### CloudFront：動態 IP 池
```
CF 的 IP 策略：
├─ IP 集合會定期變化
│  （AWS 根據負載自動調度）
├─ 用戶不應該寫死 CF 的 IP
└─ 應該透過 DNS 查詢獲得 IP

CloudFront Domain Name：
  d123.cloudfront.net  ← 用這個，不用 IP
```

**為什麼不同？**
- CDN 需要高度動態的負載均衡
- 多個邊緣節點共用一個域名
- DNS 會根據用戶地理位置返回不同 IP
- 用戶通常用域名，不需要知道 IP

### 3. BYOIP（Bring Your Own IP）詳解

#### 什麼是 BYOIP？

```
場景：你有自己的公網 IP 區間
      想要在 AWS 中使用

BYOIP 流程：
1. 準備 IP 區間
   └─ 需要在 AWS 外已有該 IP

2. 透過 AWS 驗證所有權
   └─ 需要放置特定文件驗證

3. GA 接管這些 IP
   └─ 成為全球 Anycast 入口

4. 後端任意轉移
   └─ 改 Region、改實例都行
```

#### 典型應用場景：IoT 和舊系統遷移

```
背景故事：
金融公司有 500 台舊交易終端
  → 韌體內寫死 IP：119.28.1.100
  → 無法更新程式碼
  → 現在要遷移到 AWS

傳統方案的困境：
❌ 換 Elastic IP？只能用在一個 Region
❌ 用 Route 53？DNS 無法改硬編碼的 IP
❌ 改程式碼？500 台設備，成本巨大

GA + BYOIP 解法：
┌──────────────────────────────┐
│ 119.28.1.100（企業自有 IP）  │
│       ↓ BYOIP 導入 AWS       │
│ Global Accelerator 接管      │
│       ↓                       │
│ 後端可在任何 Region 部署      │
│ ├─ us-east-1（主用）         │
│ ├─ us-west-2（備用）         │
│ └─ eu-central-1（故障轉移）  │
└──────────────────────────────┘

結果：
✅ 500 台設備 0 修改
✅ 後端隨意擴展
✅ 秒級故障轉移
✅ 成本大幅節省
```

**考題驗證**
```
題目 (ANS-C01)：
公司有 200 台 IoT 感測器，寫死了 IP 192.0.2.50。
需要遷移到 AWS，但改不了感測器韌體。
如何實現全球低延遲？

選項：
A) Elastic IP + Route 53 地理路由
B) CloudFront 作為前端
C) GA + BYOIP ✅
D) Direct Connect + VPN

答案：C
理由：
  - 只有 GA 支援靜態 IP
  - 只有 GA + BYOIP 支援企業自有 IP
  - BYOIP 是解決硬編碼 IP 的唯一方案
```

### 4. 快取機制

#### Global Accelerator：純轉發，不快取
```
GA 的流量路徑：
用戶請求
  ↓
GA 邊緣節點（檢查健康狀態，選擇最佳後端）
  ↓
直接轉發到後端應用（EC2、ALB、Lambda）
  ↓
應用回應
  ↓
GA 轉發給用戶
  ↓
❌ 沒有快取，每次都回源
```

**特徵**：
- 每次請求都打到後端（真實數據）
- 適合需要即時更新的應用
- 不會有陳舊資料問題

#### CloudFront：主動快取，邊緣儲存
```
CF 的流量路徑（首次請求）：
用戶請求（例如：/images/logo.png）
  ↓
CF 邊緣節點
  ├─ 快取中有？
  │  └─ ✅ 直接回傳（<1ms）
  └─ 快取中沒有？
     ↓
     回源到 Origin Server（你的 ALB/S3）
     ↓
     CF 邊緣快取該內容
     ↓
     回傳給用戶
```

**快取策略配置**：
```
CF 基於以下參數快取內容：
├── URL 路徑（/static/ vs /api/）
├── 查詢參數（?v=1 vs ?v=2 視為不同）
├── HTTP Headers
│   ├── Host
│   ├── Accept-Language
│   └── Cookie（可自定義）
├── 副檔名（.html、.jpg、.mp4）
└── TTL（快取時長，0-31536000 秒）

實際數據：
靜態資源（圖片、CSS、JS）
  → 快取命中率：80-95%
  → 網頁快速載入 60-80%

動態內容（API 回應）
  → 可設定 TTL=0（不快取）
  → 或短 TTL（60 秒）

影片串流
  → 可快取 1 天~永久
  → 大幅減少後端頻寬
```

**考題驗證**
```
題目 (SAP-C02)：
電商網站，靜態商品圖片頻繁被訪問，
API 回應每秒都變。應如何加速全球存取？

分析：
靜態內容（圖片） → 用 CF 快取
  ✅ 邊緣快取命中率高
  ✅ 大幅減少後端負載

動態內容（API） → 用 GA 或 CF 無快取
  ✅ GA：純路由優化
  ✅ CF 無快取模式：利用 HTTP/2 Push 等優化

最優方案：GA + CF 搭配
  ├─ CF 快取靜態資源
  └─ CF 無快取轉發動態 API
```

### 5. 故障轉移能力

#### Global Accelerator：秒級多區域故障轉移

```
GA 內部健康檢查：
┌─────────────────────────────┐
│ 每 10 秒掃描一次後端健康狀態 │
└──────────┬──────────────────┘
           ↓
┌─────────────────────────────┐
│ 檢測到故障（連線拒絕等）    │
└──────────┬──────────────────┘
           ↓
┌─────────────────────────────┐
│ 立即重定向新連線到備用端點  │
│ （<30 秒內）                │
└─────────────────────────────┘

技術細節：
- 檢查間隔：10-30 秒（可配置）
- 連續失敗次數：預設 3 次判定故障
- 故障轉移：立即開始轉向下一健康端點
- 已建立連線：無法自動切換，需用戶重連
```

**多區域故障轉移示例**：
```
場景：金融交易系統跨 3 個 Region 部署

US-EAST-1
  ├─ Primary Endpoint（主）
  ├─ ALB + 10 個 EC2 實例
  └─ 100% 流量初始

US-WEST-2
  ├─ Secondary Endpoint（備）
  ├─ ALB + 5 個 EC2 實例
  └─ 0% 流量（待命）

EU-CENTRAL-1
  ├─ Tertiary Endpoint（三級）
  ├─ ALB + 3 個 EC2 實例
  └─ 0% 流量（待命）

故障情況：
時間點 0:00   │ US-EAST-1 完全掛掉（DDoS、硬體故障）
時間點 0:15   │ GA 檢測到故障
時間點 0:20   │ 自動轉移所有新連線 → US-WEST-2
時間點 0:21   │ 業務繼續（零人工介入）

用戶體驗：
- 已連線用戶：感受到一次暫停，然後繼續
- 新連線用戶：自動連到 US-WEST-2
```

#### CloudFront：分佈式容錯，無單點故障

```
CF 的高可用架構：
┌─────────────────────────────────┐
│ 全球 500+ 邊緣節點（Edge）       │
│ + 13 個區域快取節點（Regional）  │
└────────────┬────────────────────┘
             ↓
單個邊緣節點故障
  ↓
用戶自動轉到其他邊緣節點（透明轉移）
  ↓
沒有額外延遲

如果多個節點故障
  ↓
快取層級降級：
  Edge Cache → Regional Cache → Origin
  ↓
內容仍可用，只是可能延遲增加
```

**CF 的容錯設計**：
- 無單點故障
- 自動容錯（設計上就這樣）
- 無需額外健康檢查配置
- Origin 故障時，邊緣快取可降級提供舊版本

**對比分析**：
| 指標 | GA | CF |
|------|----|----|
| **故障檢測** | 主動檢查後端 | 被動故障降級 |
| **轉移速度** | 秒級（10-30s） | 毫秒級（自動） |
| **適用場景** | 後端故障，需跨區轉移 | 邊緣節點故障（內建容錯） |

## 第三層：性能指標對比

### 延遲改進

```
改進幅度（基準 = 純公網路由）：

靜態資源場景（CF 優勢）：
  ├─ GA 路由優化：25-30% 改進
  ├─ CF 邊緣快取：60-80% 改進 ✅ 最佳
  └─ 原因：快取命中，無需回源

動態內容場景（GA 優勢）：
  ├─ GA：25-30% 改進 ✅
  ├─ CF 無快取：15-20% 改進
  └─ 原因：GA 路由優化的優勢更明顯

TCP 優化細節：
  ├─ GA Edge：在邊緣完成 TCP 握手
  ├─ GA → Origin：邊緣用骨幹網連接
  ├─ 效果：減少首字節延遲（TTFB）
  └─ 典型改進：100ms → 50ms
```

**實際案例**：
```
用戶：日本東京
應用：AWS 美國俄亥俄州（us-east-1）

純公網路由：
  東京 → 公網 ISP → 美國 ISP → 俄亥俄州
  延遲：~200ms

GA 路由：
  東京 → GA 邊緣節點（東京附近）
       → AWS 骨幹網（私有、低擁塞）
       → 俄亥俄州
  延遲：~140ms（減少 30%）

CF 快取（假設快取命中）：
  東京 → CF 邊緣節點（東京 ID）
  延遲：~5ms（減少 97%）
```

### 資料傳輸成本對比

#### Global Accelerator 定價

```
成本構成：
┌──────────────────────────────┐
│ 固定成本                     │
├──────────────────────────────┤
│ $0.025 / 小時 / 加速器       │
│ × 730 小時 / 月              │
│ = $18.25 / 月 per GA         │
└──────────────────────────────┘

┌──────────────────────────────┐
│ 變動成本（流量）             │
├──────────────────────────────┤
│ 入站：$0.02 / GB             │
│ 出站：$0.02 / GB             │
│ （根據 Region 略有差異）     │
└──────────────────────────────┘

月度成本計算（100GB 雙向流量）：
  = 固定 + 入站 + 出站
  = 18.25 + (100 × 0.02) + (100 × 0.02)
  = 18.25 + 2 + 2
  = $22.25 / 月
```

#### CloudFront 定價

```
成本構成：
┌──────────────────────────────┐
│ 無固定成本                   │
└──────────────────────────────┘

┌──────────────────────────────┐
│ 資料傳輸                     │
├──────────────────────────────┤
│ 出站：$0.085-0.20 / GB       │
│ （依 Region 和流量級別）     │
└──────────────────────────────┘

┌──────────────────────────────┐
│ 請求費用                     │
├──────────────────────────────┤
│ HTTP 請求：$0.0075/10000     │
│ HTTPS 請求：$0.0100/10000    │
└──────────────────────────────┘

月度成本計算示例（假設出站 $0.085/GB）：
100GB 流量 + 1000 萬 HTTPS 請求
  = (100 × 0.085) + (1000 × 0.01)
  = 8.50 + 10
  = $18.50 / 月
```

#### 成本對比決策

```
流量級別對比（假設數據傳輸 GA $0.02/GB, CF $0.085/GB）：

月流量 50GB：
  GA：18.25 + (50 × 0.04) = $20.25
  CF：50 × 0.085 = $4.25
  ✅ CF 便宜 80%

月流量 100GB：
  GA：18.25 + (100 × 0.04) = $22.25
  CF：100 × 0.085 = $8.50
  ✅ CF 便宜 62%

月流量 500GB：
  GA：18.25 + (500 × 0.04) = $38.25
  CF：500 × 0.085 = $42.50
  ✅ GA 便宜 10%

月流量 1000GB：
  GA：18.25 + (1000 × 0.04) = $58.25
  CF：1000 × 0.085 = $85.00
  ✅ GA 便宜 32%

對比結論：
  - 月流量 <200GB → CF 更便宜（適合小型應用）
  - 月流量 >500GB → GA 相對便宜（適合大型應用）

快取效應的重要性：
  - CF 邊緣快取可減少出站流量 50-80%
  - 快取友善的應用，CF 成本會更優
  - 例：100GB 入站，快取命中 70%
    → 實際出站只需 30GB
    → CF 成本：30 × 0.085 ≈ $2.55（大幅降低）
```

## 第四層：應用場景決策

### 決策樹

```
我的應用需要加速，應該選 GA 還是 CF？

【第一級】協議檢查
│
├─ 使用 TCP/UDP（非 HTTP）？
│  ├─ 是 → GA ✅（CF 不支援）
│  └─ 否 → 繼續
│
├─ 使用 HTTP/HTTPS？
│  ├─ 是 → 繼續
│  └─ 否 → GA ✅
│
【第二級】硬編碼 IP 檢查
│
├─ 用戶端寫死了 IP？
│  ├─ 是 → GA + BYOIP ✅
│  └─ 否 → 繼續
│
【第三級】內容特性檢查
│
├─ 主要是靜態資源（圖片、影片、CSS）？
│  ├─ 是 → CF ✅（快取效率最高）
│  └─ 否 → 繼續
│
├─ 主要是實時 API 資料？
│  ├─ 是 → GA ✅（無快取，即時性）
│  └─ 否 → 繼續
│
├─ 混合型（靜態+動態）？
│  ├─ 是 → GA + CF 搭配 ✅
│  └─ 否 → 繼續
│
【第四級】故障轉移需求
│
├─ 需要跨 Region 秒級故障轉移？
│  ├─ 是 → GA ✅
│  └─ 否 → 繼續
│
【第五級】成本考量
│
└─ 月流量多少？
   ├─ <200GB → CF ✅
   └─ >500GB → GA ✅
```

### 典型應用場景

#### 場景 1：全球電商平台
```
需求：
  ✓ 全球用戶快速載入商品列表和圖片
  ✓ 購物車、支付 API 低延遲
  ✓ 後端在 3 個 Region（美東、歐洲、新加坡）

流量構成：
  ├─ 靜態資源（圖片、CSS、JS）：70%
  └─ 動態 API：30%

最優方案：GA + CF 搭配

架構：
┌──────────────────────────────┐
│ CloudFront                   │
│ ├─ 快取靜態資源              │
│ ├─ 快取命中率：80-85%        │
│ └─ 減少後端流量：60-70%      │
└──────────┬───────────────────┘
           │ Miss：回源到 Origin
           ↓
┌──────────────────────────────┐
│ Global Accelerator           │
│ ├─ 路由動態 API 請求         │
│ ├─ 秒級故障轉移              │
│ └─ 3 Region 高可用           │
└──────────┬───────────────────┘
           ↓
┌──────────────────────────────┐
│ 後端應用                     │
│ ├─ us-east-1 (主)            │
│ ├─ eu-central-1 (備)         │
│ └─ ap-southeast-1 (備)       │
└──────────────────────────────┘

成本估算（月 500GB 流量）：
  CF：  500 × 0.085 = $42.50
        但快取減少出站 70% → 實際 $12.75
  GA：  18.25 + (150 × 0.04) = $24.25
  總計：$37 / 月
```

#### 場景 2：即時遊戲伺服器
```
需求：
  ✓ 全球玩家毫秒級連線
  ✓ 遊戲狀態 UDP 通訊
  ✓ 聊天系統 TCP 連線
  ✓ 多 Region 部署，一個宕機時秒級轉移

協議分析：
  ├─ 遊戲狀態：UDP（CF 不支援 ❌）
  ├─ 聊天系統：TCP（CF 支援但無法優化 ⚠️）
  └─ GA 完美支援所有 ✅

最優方案：GA 獨自承擔（不需要 CF）

理由：
  ❌ CF 無法快取遊戲狀態（高度動態）
  ❌ CF 無法處理 UDP
  ✅ GA 直接、高效

架構：
             ┌─ us-east-1
使用者群 ──→ GA ─┼─ eu-west-1
             └─ ap-northeast-1
             （秒級轉移）

成本特徵：
  ├─ GA 固定成本：$18.25/月
  ├─ GA 流量成本：100GB × $0.04 = $4
  └─ 總計：$22.25/月

  （沒有 CF 的 $0.0075/請求）
```

#### 場景 3：IoT 感測器網路（硬編碼 IP 遷移）
```
背景：
  工廠有 500 台感測器，韌體寫死 IP：192.0.2.100
  現在要遷移到 AWS，但改不了設備

需求：
  ✓ 感測器無感知地連到 AWS
  ✓ 後端高可用，支援故障轉移
  ✓ MQTT 協議（UDP/TCP）

唯一解法：GA + BYOIP

流程：
1. BYOIP 驗證：將 192.0.2.100 導入 AWS
2. GA 接管：作為全球 Anycast 入口
3. 後端任意部署：可在任何 Region
4. 故障轉移：秒級自動切換

架構：
┌──────────────────────────────┐
│ 192.0.2.100（企業自有 IP）    │
│ ↓ BYOIP 驗證                 │
│ Global Accelerator 接管      │
│ ├─ us-east-1 Lambda          │
│ ├─ us-west-2 Lambda（備）    │
│ └─ eu-central-1 Lambda（備）│
└──────────────────────────────┘

效果：
  ✅ 500 台設備零修改
  ✅ 秒級故障轉移
  ✅ 後端可隨意升級
  ✅ 成本控制（小流量）

成本（月 10GB MQTT 流量）：
  GA：18.25 + (10 × 0.04) = $18.65/月
```

## 第五層：AWS 考試重點分析

### 高頻考題類型

#### Type A：協議與適用性判斷
```
常見出題形式：
「應用使用 [特定協議]，需要 [目標]。應選什麼？」

典型題目（SAP-C02）：
遊戲公司需要用 UDP 傳輸遊戲狀態，全球玩家延遲 <100ms。

A) CloudFront + Route 53
B) Global Accelerator ✅
C) ALB + Route 53 地理路由
D) ELB

答案解析：
  ✅ B 是唯一支援 UDP 的 AWS 加速服務
  ❌ A：CF 不支援 UDP
  ❌ C/D：無法優化跨洲延遲
```

#### Type B：BYOIP 與硬編碼 IP
```
常見出題形式：
「舊系統寫死了 IP，現在要遷移...應如何解決？」

典型題目（ANS-C01）：
銀行有 200 台 ATM，寫死了連線 IP 203.0.113.50。
需要遷移到 AWS，但改不了 ATM 韌體。
如何實現故障轉移和全球負載均衡？

A) Elastic IP + Route 53
B) CloudFront + DNS failover
C) GA + BYOIP ✅
D) Direct Connect + BGP

答案解析：
  ✅ C 是唯一完整解決方案
     - GA 提供靜態 IP（Elastic IP 只支援單 Region）
     - BYOIP 支援企業自有 IP
     - 秒級故障轉移
  ❌ A：EIP 無法跨 Region 故障轉移
  ❌ B：DNS 無法改寫硬編碼 IP
  ❌ D：DC 無法接管 IP，太複雜
```

#### Type C：內容類型與服務匹配
```
常見出題形式：
「要加速 [內容類型]，並降低成本...」

典型題目（SAP-C02）：
視訊平台需要分發 4K 影片到全球，流量 1000GB/月。
如何最節省成本？

A) 用 GA 加速
B) 用 CF 快取 ✅
C) 用 S3 跨 Region 複製
D) 用 VPN 連接

答案解析：
  ✅ B 最優：CF 邊緣快取大檔案
     - 影片：天生適合快取（可快取數天~永久）
     - 1000GB 月流量：CF $0.085/GB = $85
     - GA：$18.25 + $40 = $58.25（費用高出）
  ❌ A：不需要 GA，CF 更便宜
  ❌ C：複雜且昂貴
  ❌ D：沒有提升效能
```

#### Type D：混合架構設計
```
常見出題形式：
「應用既有靜態資源又有動態 API，應如何架構？」

典型題目（SAP-C02）：
SaaS 應用在 3 個 Region 部署。
靜態資源（40%）來自 S3，API（60%）來自 Lambda。
需要全球低延遲和故障轉移。

最優方案是什麼？

A) 只用 GA
B) 只用 CF
C) GA + CF 搭配 ✅
D) Route 53 + CloudWatch

答案解析：
  ✅ C 是最優設計：
     CF 快取靜態 → 80-95% 命中率
     GA 路由 API → 直達後端
     各司其職，發揮各自優勢

  ❌ A：GA 無快取，無法優化靜態資源
  ❌ B：CF 無法快取 API，API 延遲高
  ❌ D：Route 53 無法替代加速服務
```

### 考試金句速記

```
「3 句話秒殺 GA vs CF」：

1. UDP / TCP / 非 HTTP？
   → 只有 GA

2. 硬編碼 IP / BYOIP？
   → 只有 GA

3. 靜態內容 / 快取友善？
   → 通常 CF

4. 跨 Region 故障轉移？
   → GA（秒級）

5. 小流量節省成本？
   → CF（無固定費）

6. 大流量？
   → GA（相對便宜）
```
