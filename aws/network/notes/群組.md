【觀念釐清：Target Group vs. Placement Group】
把它們想像成開一間餐廳：

🎯 Target Group (目標群組) = 「服務生端盤子的目標」
這是什麼： 它是跟 Load Balancer (負載平衡器, ALB/NLB) 綁定在一起的邏輯群組。

功用： 當客戶點餐（網路請求進來），服務生（Load Balancer）要看 Target Group 裡面有哪幾台 EC2 伺服器（廚師）是健康的，然後把訂單派給他們。它完全不管這些機器實體上放在哪裡。

考場反射關鍵字： Load Balancer, ALB/NLB, Routing traffic, Health Checks。

🏢 Placement Group (放置群組) = 「廚師們的實體座位安排」
這是什麼： 這是用來控制 EC2 實例在 AWS 機房底層**「實體硬體 (Hardware/Rack)」**上的排列方式。

考場反射關鍵字： Physical hardware, Low latency, High network throughput, Hardware failure isolation。

2. 【Placement Group 的三大陣型 (考試必考！)】
在 AWS 裡，排座位有三種策略，這題考的是第一種：

Cluster (叢集)： 「大家擠在一起」。把所有 EC2 塞在同一個實體機櫃裡。

優點： 網路延遲超低、頻寬超大 (10 Gbps)。

考點： 適合 HPC (高效能運算)、大數據分析。

Spread (分散)： 「大家離越遠越好」。強迫每一台 EC2 放在「不同的底層實體硬體機櫃」上。

優點： 絕對的高可用性。一個機櫃燒掉，只會死一台機器。

考點： 每個 AZ 最多只能放 7 台（因為 AWS 找不到那麼多獨立機櫃給你）。適合極度關鍵的少數伺服器。

Partition (分區)： 「分組隔離」。介於兩者之間，把機器分成幾個邏輯區塊（Partition），不同區塊在不同機櫃。適合 Hadoop, Cassandra。

3. 【題幹手術：為什麼會跳 Capacity Error？為什麼選 B？】
現在我們回到題目。你一開始開了 9 台 EC2 在一個 Placement Group（通常這種情境是指 Cluster 叢集型），一開始跑得很順。後來你想再加 2 台，結果跳出 Capacity Error (容量不足錯誤)。

為什麼 C 是錯的？
AWS 並沒有「一個 Placement Group 最多只能放 10 台」這種死規定。所以 C 選項純粹是瞎掰的數字。

為什麼會發生 Capacity Error？
想像你在餐廳訂了一個 9 人桌 (Cluster 機櫃)。大家都坐下了。現在來了 2 個朋友，你想加椅子，但這張桌子（實體機架）旁邊的空間剛好被別桌的客人佔滿了！AWS 的底層硬體剛好沒有連續的空位可以塞下這 2 台新機器。

🎯 為什麼 B (Stop and restart) 是官方的最佳解法？
這是一個非常經典的 AWS 實務 Best Practice。既然這張 9 人桌擠不下 11 個人，你要怎麼做？
解法： 請這 9 個人全部「站起來 (Stop instances)」，然後請餐廳經理（AWS）直接幫你們找一張「可以同時容納 11 個人的全新大桌子」，然後大家一起「重新坐下 (Start instances)」。
當你 Stop 並 Start 這些實例時，AWS 會重新在整個機房裡尋找一塊足夠大的實體硬體空間，把這 11 台機器一起放進去。

看到 Placement Group + Capacity Error 👉 唯一解法就是 Stop and Start all instances (全部關機再開機)，讓 AWS 幫你換個大機櫃。

1. 🎯 Target Group (目標群組)：流量分配的集散地
【實務上怎麼用？】
只要你用了 Load Balancer (ALB 或 NLB)，你百分之百會用到 Target Group。它就像是負載平衡器後面的「值班表」。
實務上，我們會把 Web 伺服器 (EC2)、容器 (ECS Tasks) 甚至是無伺服器函數 (Lambda) 放進 TG 裡面。當使用者的 HTTP 請求來到 ALB，ALB 就會看 TG 裡誰醒著，把流量派給它。

【實務與考試的痛點】

痛點一：Health Check (健康檢查) 失敗地獄

實務血淚： 你明明看到 EC2 開機了，但 TG 卻顯示 Unhealthy，流量完全進不去。

原因： 通常是 Security Group (安全組) 忘記開放 ALB 的 IP 進來，或者是你設定的健康檢查路徑（例如 /health）沒有寫好，伺服器回傳了 404 Not Found 而不是 200 OK。

痛點二：Deregistration Delay (取消註冊延遲 / 以前叫 Connection Draining)

實務血淚： 伺服器要縮編 (Scale in) 關機時，正在結帳的客戶突然斷線！

原因： TG 預設會給 300 秒的「緩衝期」。如果你把它設太短，ALB 會立刻把該台機器踢掉，不管上面還有沒有人在連線。

2. 🏢 Placement Group (放置群組)：底層硬體的風水寶地
【實務上怎麼用？】
平常架設一般網站，我們根本不會去設定 PG，就讓 AWS 隨便幫我們找機器放就好。PG 只有在面對**「極端需求」**時才會拿出來用：

Cluster (叢集)： 公司在算 AI 模型、跑大數據，幾十台機器需要互相瘋狂傳資料。把它們塞在同一個機櫃，享受 10Gbps 以上的超低延遲內網。

Spread (分散)： 公司有 3 台極度重要的資料庫 (如 MySQL Galera Cluster)。為了怕機房停電或機櫃燒毀，強制把它們分散在「不同實體電源和網段的機架上」。

【實務與考試的痛點】

痛點一：容量不足 (Capacity Error) - 就是你截圖考的這題！

實務血淚： Cluster PG 就像是訂了一間 KTV 包廂，一開始 9 個人剛剛好，後來想加 2 個人，結果包廂滿了（實體機架沒空間）。

解法： 只能把所有人趕出來（Stop instances），請經理換一間大包廂，再重新進去（Start instances）。

痛點二：Spread PG 的「7 台魔咒」

實務血淚： 你想把 10 台機器做硬體分散，結果 AWS 跳錯。

原因： 這是考試極愛考的限制：Spread PG 在每一個 AZ（可用區）最多只能放 7 台機器！ 因為 AWS 要保證給你獨立的機架，資源有限。

💡 教練的實戰總結
TG (Target Group) 是軟體/網路層的概念： 痛點通常圍繞在「流量進不進得來」和「機器健不健康」。

PG (Placement Group) 是實體/硬體層的概念： 痛點通常圍繞在「空間夠不夠放」和「硬體數量的極限」。
