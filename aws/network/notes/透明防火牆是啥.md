這題的核心是在考 AWS 近幾年最強大的網路資安架構：「透明防火牆 (Bump-in-the-wire)」 以及 「Gateway Load Balancer (GWLB)」 的運作機制。

我們先把題目忘掉，我來為你建構這個必備知識：

1. 什麼是「透明防火牆 (Bump-in-the-wire)」？
在傳統網路中，如果你要在伺服器前面擋一台防火牆，通常有兩種作法：

Proxy (代理) 模式： 封包先送到防火牆的 IP，防火牆再代替你發給伺服器。缺點：伺服器看到的來源 IP 都是防火牆的 IP，抓不到真實的駭客 IP。

NAT 模式： 防火牆做位址轉換。缺點：架構變得很複雜。

「透明防火牆 (Bump-in-the-wire)」 顧名思義，就是「線路上的隱形攔截點」。

情境： 封包從外面進來，目標 IP 明明寫著你的 EC2。但在抵達 EC2 的半路上，封包被「偷偷」吸到旁邊的防火牆檢查。

重點： 檢查完放行後，封包繼續走向 EC2。EC2 完全不知道封包被檢查過，駭客也不知道中間有防火牆，雙方看到的 IP 都是最原始的。 這就是「透明」。

2. AWS 如何實現透明防火牆？ (三大神器)
要做到這件事，AWS 發明了三個互相配合的元件，這也是解題的關鍵：

神器一：Gateway Load Balancer (GWLB)
這是一台專門用來乘載「第三方防火牆 (Palo Alto, Fortinet)」的負載平衡器。

它有一個黑科技叫 GENEVE 封裝。當封包進來時，它「不會」竄改原本的來源/目的 IP，而是像包禮物一樣，把整個原始封包包起來，丟給後面的防火牆檢查。防火牆檢查沒問題，GWLB 就把包裝拆掉，讓封包繼續走。

神器二：GWLB Endpoint (GWLBE) ——「傳送門」
這是一個建在你 Application VPC 裡面的虛擬端點。

因為你的防火牆通常集中放在一個獨立的「資安 VPC (Security VPC)」裡以便統一管理。

這個 GWLBE 就像一個 「跨 VPC 傳送門」。你只要把封包丟進 GWLBE，封包就會瞬間傳送到隔壁資安 VPC 的 GWLB 裡面。

神器三：Gateway Route Table (閘道路由表) ——「門口的交通警察」
這是最關鍵的一步！我們稱之為 Ingress Routing (入口路由)。

過去，Internet Gateway (IGW) 收到網際網路來的封包，只會傻傻地直接送進 VPC 給 EC2。

現在，你可以把一張 Route Table「綁定在 IGW 身上」。

你可以在這張表上規定：「所有從網際網路進來、準備要去 EC2 網段的封包，通通給我先轉彎去 GWLBE (傳送門)！」

3. 完整的封包奇幻漂流 (透明攔截流程)
理解了這三個神器，我們來看封包是怎麼被透明攔截的：

駭客 (外網) 發出攻擊封包： 目標 IP 寫著你的 EC2 公有 IP。

抵達 IGW (大門口)： IGW 看了身上的 Gateway Route Table (閘道路由表)，發現規則寫著：「去 EC2 之前，先去 GWLBE」。

進入傳送門： 封包被吸進 GWLBE，瞬間傳送到隔壁 VPC 的 GWLB。

防火牆檢查： GWLB 把封包丟給 Fortinet/Palo Alto 防火牆檢查。防火牆說：「沒問題，放行。」

回到傳送門： 封包從 GWLBE 吐出來。

抵達目標： 封包終於順利抵達 EC2。EC2 看到的來源 IP 依然是駭客的真實 IP。

(回程的封包也是一樣，透過 EC2 所在的 Subnet Route Table 規定 0.0.0.0/0 先丟給 GWLBE，再出去給 IGW。)

回頭看第 153 題的解析
現在你具備了上帝視角，我們來看題目：

現狀： EC2 在 Application VPC。資安工程師已經在另一個 Standalone VPC (獨立 VPC) 建好了 GWLB 與防火牆艦隊。

任務： 你要讓來自 Internet 的流量「先」被防火牆檢查，才能抵達 EC2。

來檢驗選項：

C 選項：

在 Application VPC 建一個 GLB Endpoint (傳送門)。

建立一張 Gateway Route Table 並綁在 IGW 上，設定目標為 EC2 網段的封包，下一跳是 GLB Endpoint。

更新 EC2 的 Subnet Route Table，讓回程流量 (預設路由) 也指向 GLB Endpoint。

這完全就是我們上面教的「透明防火牆」標準施工步驟！

為什麼 D 是錯的？

D 叫資安工程師把 GWLB「搬進 (move)」Application VPC。

在企業架構中，防火牆放在獨立的 Security VPC 是為了集中管理與合規（可能要服務十幾個不同的 App VPC）。把它搬進單一 App VPC 破壞了這個架構。有了 GWLB Endpoint 這個傳送門技術，我們根本不需要把實體設備搬來搬去。

核心原則

GLB 的 Ingress Routing 模式 = Gateway Route Table on IGW + GLB Endpoint in same VPC，這是 AWS 官方標準架構


GLB Endpoint 是「門」，GLB 本體是「檢查站」

用現實類比理解
你家社區 = Application VPC
社區大門 = GLB Endpoint  ← 必須在你家社區內
保全總部 = GLB 本體（standalone VPC）

保全總部不可能搬進每個社區，但每個社區**必須有一個對講機（Endpoint）**連回總部。


技術原因
GLB 使用的是 AWS PrivateLink 架構：
Application VPC          Standalone VPC
┌─────────────┐          ┌──────────────────┐
│             │          │                  │
│ GLB Endpoint│◄────────►│ GLB + Firewalls  │
│ (像插座)    │PrivateLink│ (像發電廠)       │
│             │          │                  │
└─────────────┘          └──────────────────┘

GLB 本體：集中管理 firewall fleet，部署一次即可
GLB Endpoint：每個需要檢查的 VPC 各放一個，作為「本地接入點」
流量不需要跨 VPC 路由，直接透過 PrivateLink 私有傳輸
