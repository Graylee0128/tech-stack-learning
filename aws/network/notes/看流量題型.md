# AWS 網路流量檢查題型

## 題目
Your company has an EC2 Instance hosted in AWS. This EC2 Instance hosts an application. Currently this application is experiencing a number of issues. You need to inspect the network packets to see what the type of error that is occurring? Which one of the below steps can help address this issue.

**選項：**
- A. 使用 EC2 Instance 上的原生網卡工具 (tcpdump 等)
- B. 使用 AWS Partner (合作夥伴) 提供的網路監控工具
- C. 在 EC2 實例的網卡開啟 Promiscuous Mode
- D. 查看 CloudWatch Metrics

---

## 分析思路

### 1️⃣ 核心需求：深度封包檢測 (DPI)

題目明確說了需要 **"inspect the network packets"**（檢查網路封包）來找出 Error 的類型。這就是 **Deep Packet Inspection (DPI, 深度封包檢測)**，需要把封包拆開來看裡面的 Payload（真實資料）。

### 2️⃣ 選項分析

#### ❌ 選項 A：EC2 原生工具（tcpdump）
- **問題**：在 AWS 中，由於 VPC 的網卡虛擬化特性，直接在 EC2 上跑 tcpdump 只能看到本機發送的流量，無法捕獲端對端的完整流量
- **場景限制**：只適用於本地診斷，不適合完整的流量檢查

#### ❌ 選項 C：Promiscuous Mode（混雜模式）
- **AWS 絕對禁忌**：在 AWS VPC 中，網卡嚴禁開啟混雜模式
- **原因**：混雜模式會嘗試竊聽其他 EC2 實例的流量，違反 AWS 多租戶隔離政策
- **後果**：AWS 直接阻止此操作，網卡會被禁用

#### ❌ 選項 D：CloudWatch Metrics
- **本質**：只是「儀表板數字」（Network In/Out、CPU 使用率等）
- **局限**：用來看「效能趨勢」，完全沒有封包內容
- **不適合用途**：無法看出封包裡發生了什麼 HTTP Error

#### ✅ 選項 B：AWS Partner 網路監控工具（舊方案）

**作法**：
1. 在 AWS Marketplace 購買第三方虛擬防火牆（Virtual Appliance）
   - 如：Palo Alto、Cisco、Check Point
2. 將其部署在 EC2 前面（作為中間層）
3. 讓它幫你進行深度封包檢測和分析

---

## 現代方案：AWS VPC Traffic Mirroring（推薦）

💡 **AWS 官方後來推出的原生解決方案**

### 核心概念
在網卡 (ENI) 層級開啟「流量鏡像」功能，自動將通過該網卡的真實封包複製一份，送至監控工具進行分析。

### 優勢
- ✅ 完全原生，無需第三方工具
- ✅ 無需開啟混雜模式
- ✅ 支援深度封包檢測 (DPI)
- ✅ 可與 Wireshark、Suricata 等工具集成
- ✅ AWS 官方推薦的最佳實踐

### 使用流程
1. 在 EC2 網卡 (ENI) 上配置 Traffic Mirror
2. 指定監控工具的目標（同 VPC 內的另一個 ENI 或專用監控 EC2）
3. 監控工具接收鏡像流量進行分析
