在絕大多數的 AWS 考試與實務場景中，修改 DHCP Option Set 被視為「非必要不建議」的作法。在 AWS 的架構設計哲學裡，Route 53 Resolver（也就是你看到的 Inbound/Outbound Endpoint）已經取代了修改 DHCP Option Set 成為處理混合雲 DNS 的「黃金標準」。為什麼不建議動 DHCP Option Set？1. 會失去「雲端原生」的解析能力這是最致命的一點。AWS 預設的 DHCP 會指向 AmazonProvidedDNS（即 VPC 網路範圍的 .2 位址）。優點： 它能自動解析 Private Hosted Zones (PHZ)、VPC 內部的 DNS Hostnames 以及 AWS 服務終端節點。後果： 如果你改了 DHCP，強行指向地端 DNS 或特定伺服器，你的 EC2 就會失去對這些 AWS 原生資源的解析能力，除非你在地端 DNS 手動維護複雜的轉發邏輯。2. 生效緩慢且具侵入性DHCP： 當你更改 VPC 的 DHCP Option Set 時，已經在執行的 EC2 實例不會立即生效。它們必須等到 DHCP 租約到期並續約，或者你需要手動重啟網路服務（甚至重啟機器）才能抓到新設定。Resolver Rule： 這是網路層級的「透明攔截」。只要規則一關聯，流量立刻生效，EC2 完全不需要做任何變更。3. 維運複雜度高在多 VPC 環境下，如果你每個 VPC 都去改 DHCP，當你地端 DNS IP 變更時，你要逐一修改所有 VPC 的設定。而使用 Route 53 Resolver，你只需要在中央（如 Shared Services VPC）改一次 Outbound Endpoint 或 Rule，所有關聯的 VPC 就同步更新了。什麼時候「才需要」改 DHCP Option Set？雖然不推薦，但在以下特殊實務場景，你可能還是得動它：Active Directory (AD) 強烈依賴： 當你的 EC2 必須加入 Windows 網域，且該網域要求實例的 Primary DNS 必須「直接」指向 Domain Controller (DC) 以維持網域穩定性時。無法修改的遺留系統 (Legacy)： 某些老舊軟體寫死了必須從 DHCP 獲取特定的 domain-name 或特定參數（如 NTP 伺服器）。非典型的第三方設備： 某些特殊的網路虛擬設備（Appliance）不支援透過 .2 進行轉發。總結：解題與實務的「判斷準則」在畫架構圖或考試時，你可以參考下表來快速做決定：需求場景推薦做法為什麼？AWS 查地端名稱Route 53 Outbound Resolver透明轉發，保留 AWS 原生解析能力。地端查 AWS 名稱Route 53 Inbound Resolver提供靜態 IP 讓地端 DNS 做轉發。跨 VPC 共享 DNSResolver Rules + RAM 共享集中管理，最少配置成本。強行指定 DNS ServerDHCP Option Set最後手段，通常是為了相容性或特定協議。一言以蔽之： 在 AWS 的世界裡，能用「網路層轉發（Resolver Rule）」解決的事情，就不要去動「作業系統層設定（DHCP）」。
