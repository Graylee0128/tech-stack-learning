# AWS 網路效能優化筆記

## 概述
AWS 環境下的網路效能優化策略，特別是在 VPC Peering、跨 AZ 通訊、以及高效能運算（HPC）場景下的架構設計。

---

## 🏎️ AWS 網路效能的「黃金三角」

### 快速記憶

當題目要求 **"Optimal Performance"（最佳效能）**，特別是在兩個 Peered VPC 之間，腦中要立刻浮現：

1. **同可用區 (Same AZ)**
2. **放置群組 (Placement Group)**
3. **增強型網路 (Enhanced Networking)**

---

## 📊 效能優化方案對比

### B. Same AZ + Cluster Placement Group

**為什麼是最佳選擇？**

| 特性 | 效果 |
|------|------|
| **同可用區 (Same AZ)** | 物理距離最近 → 光纖傳輸距離短 → 延遲天生最低 |
| **Cluster Placement Group** | 機器放在同一機架或相鄰插槽 → 非阻塞頻寬 10-100Gbps |
| **VPC Peering + Same AZ** | 即使 VPC 不同、跨帳號，同 AZ 內依然享受低延遲優勢 |

**延遲優勢：** ⭐⭐⭐⭐⭐

### C. Enhanced Networking

**為什麼強大？**

| 特性 | 效果 |
|------|------|
| **SR-IOV 技術** | 虛擬機直接與網卡溝通，繞過 Hypervisor |
| **PPS 提升** | 每秒封包數 (Packets Per Second) 大幅提高 |
| **Jitter 降低** | 網路抖動極大降低，延遲更穩定 |
| **CPU 開銷降低** | 不再浪費 CPU 處理網路封包轉發 |

**適用場景：** HPC（高效能運算）、數據分析、即時交易系統

### D. MTU 1500（❌ 錯誤）

**為什麼不選它？**

```
1500 MTU  = 一般貨車（兼容性好、效率普通）
9001 MTU  = 大貨聯結車（VPC Peering 支援、載貨量大）

題目要求「最佳效能」→ 應選 9001 MTU (Jumbo Frames)
```

設置 1500 MTU 實際上是**把效能閹割了**。

## 💡 考場秒殺表

當看到以下關鍵字時，快速判斷：

| 題目關鍵字 | 正確選項 | 核心理由 |
|----------|--------|--------|
| **Lowest Latency（最低延遲）** | Same AZ | 物理距離最短 |
| **High Bandwidth（高頻寬）** | Cluster Placement Group | 網卡匯流排優化 |
| **Highest PPS（最高封包數）** | Enhanced Networking (ENA) | 硬體直達加速 |
| **Optimal/Best Performance（最佳效能）** | Jumbo Frames (9001 MTU) | 單次載貨量最大 |

## 🔧 Enhanced Networking 深度解析

### 傳統模式 vs 增強模式

#### 傳統網路（VIF 模式）- 「秘書傳話」

```
EC2 虛擬機
  ↓
Hypervisor（虛擬化層軟體）← 瓶頸（忙碌的秘書）
  ↓
實體網卡

問題：
- 軟體開銷（CPU Overhead）高
- 延遲大
- 網路抖動（Jitter）明顯
```

#### Enhanced Networking（SR-IOV）- 「專線直連」

```
EC2 虛擬機
  ↓
Virtual Function（硬體虛擬切分） ← 直達通道
  ↓
實體網卡

優勢：
✅ 跳過 Hypervisor
✅ 直接硬體通訊
✅ 零（或極低）軟體開銷
```

### 具體成果

| 指標 | 改善幅度 |
|------|--------|
| **PPS (每秒封包數)** | 暴增 ⚡ |
| **Latency (延遲)** | 變穩、抖動極低 |
| **CPU 使用率** | 大幅降低 |
| **吞吐量 (Throughput)** | 接近線性提升 |

---

## 📝 考試必背 3 個關鍵字

### 1. **SR-IOV**
- Enhanced Networking 的核心技術
- 全名：Single Root I/O Virtualization
- 理解：讓虛擬機直接訪問實體硬體的一部分

### 2. **Higher PPS / Lower Latency**
- Enhanced Networking 能帶來的直接好處
- 考題通常這樣強調效能提升

### 3. **驅動程式（Driver）**

| 驅動程式 | 適用機型 | 速度上限 | 時代 |
|---------|--------|--------|------|
| **ENA (Elastic Network Adapter)** | C5, M5, R5, C6, M6 等新機型 | 100 Gbps | 現代 ✅ |
| **Intel 82599 (ixgbevf)** | C3, R3, I2 等老機型 | 10 Gbps | 過時 |

## 🛠️ 實務判斷：有沒有開啟 Enhanced Networking？

**重要提示：** 選了支援的機型 ≠ 自動開啟

必須檢查兩件事：

### 1. AMI 是否支援
```
✓ 確認作業系統有沒有裝 ENA 驅動程式
✓ 使用 AWS 提供的標準 AMI 通常已包含
✓ 自定義 AMI 需要手動安裝
```

### 2. Instance 屬性是否開啟
```bash
# CLI 檢查方式
aws ec2 describe-instances --instance-ids i-xxxxx \
  --query 'Reservations[0].Instances[0].EnaSupport'

# 結果應為 true（非 false）
```

## 📌 考試應對策略

### 秒殺邏輯

看到以下場景，立刻選擇方案：

```
❓ 「如何優化網路效能」→ Enhanced Networking
❓ 「每秒封包處理量不足」→ Enhanced Networking
❓ 「降低延遲到最低」→ Same AZ + Placement Group
❓ 「最大化頻寬利用」→ Jumbo Frames (9001 MTU)
```

### 陷阱題警惕

- **不要被「老驅動」迷惑** - 題目提 Intel 82599 = 暗示過時的方案
- **MTU 1500 看起來通用，但效能題它就是錯** - VPC Peering 支援 9001
- **Enhanced Networking 需要 AMI 支援** - 光選機型還不夠

## 💬 專家碎碎念

### 為什麼這三個必須打包組合？

在現實中：
- **Same AZ + Placement Group** = 網路延遲最低
- **Enhanced Networking** = 封包處理量最高
- **Jumbo Frames** = 每個封包載貨量最大

三者合體 = **完整的超高效能方案**

### 常見面試題

> Q: 為什麼 EC2 選了支援 ENA 的機型還是沒有效能提升？
> A: 檢查 enaSupport 屬性和 AMI 驅動程式安裝情況，兩個都要開啟才行。

## 📌 快速檢查清單

- [ ] 題目強調「最低延遲」嗎？→ 優先考慮 Same AZ
- [ ] 題目提到「每秒封包數」或「PPS」嗎？→ 選 Enhanced Networking
- [ ] 題目提到「兩個 VPC 之間」嗎？→ 考慮 Peering + Placement Group
- [ ] MTU 選項出現了嗎？→ 優先選 9001（Jumbo Frames）而非 1500
- [ ] 提到「老機型」（C3、R3）嗎？→ 可能是干擾選項
