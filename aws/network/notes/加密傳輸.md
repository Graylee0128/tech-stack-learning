💡 AWS 傳輸加密題型 Cheat Sheet (看到關鍵字秒殺)
在考題中看到「VPN」、「Direct Connect」、「Encryption in Transit」時，請立刻在大腦載入這四條鐵則：

鐵則 1：VPN 永遠是「點對點 (Gateway-to-Gateway)」，絕不是「端到端 (End-to-End)」
陷阱關鍵字： End-to-end encryption, End-to-end authentication

原理： AWS Site-to-Site VPN (IPSec) 的加密隧道只存在於 Customer Gateway (客戶端閘道器) 和 VPN Gateway (VGW, AWS 端閘道器) 之間。

刪除策略： 只要題目是問「IPSec VPN 提供了什麼」，看到 End-to-end 直接刪掉！因為從地端 Server 到 CGW，以及從 VGW 到 EC2 Instance 這兩段內網傳輸，預設是明文的（除非你在應用層自己架 HTTPS/TLS）。

鐵則 2：IPSec 的神聖三位一體 (CIA)
只要考到 IPSec 的功能，它一定提供以下三個針對「這條跨網際網路隧道」的保護：

機密性 (Confidentiality)： 透過加密 (Encryption)，也就是保護傳輸中的資料 (Protection of data in transit)。

完整性 (Integrity)： 確保封包在網際網路上沒被竄改 (Data integrity protection)。

身分驗證 (Authentication)： 這裡指的是**「設備對設備 (Peer)」的驗證，確認這台 CGW 真的有資格連到這台 VGW，而不是**驗證「使用者 (Identity/User)」。

鐵則 3：Direct Connect (DX) 預設「不加密」
考點： 題目如果說「需要加密地端到 AWS 的連線」，選項如果有純 Direct Connect，直接打叉。

正確解法： 必須是 VPN over Direct Connect 或是 MACsec (針對 10Gbps/100Gbps 專線)。

鐵則 4：Client VPN vs. Site-to-Site VPN
Client VPN (OpenVPN)： 認證的是「人/設備」(User Identity, AD, SAML)。

Site-to-Site VPN (IPSec)： 認證的是「路由器/閘道器」(Peer Identity)。

🎯 套用 Cheat Sheet 拆解你的這張截圖
題目問：建立 IPSec VPN 隧道 (終止於 VGW 和 CGW 之間) 可以達成哪些目標？(選 4 個)

❌ A. End-to-end protection of data in transit: 觸發【鐵則 1】，VPN 不是 End-to-end，刪除。

❌ B. End-to-end Identity authentication: 觸發【鐵則 1】與【鐵則 4】，VPN 不做端到端，也不做終端使用者身分認證，刪除。

✅ C. Data encryption across the Internet: 觸發【鐵則 2】，IPSec 確實將跨越網際網路的這段資料加密。

✅ D. Protection of data in transit over the Internet: 觸發【鐵則 2】，與 C 幾乎是同義詞，保護跨網際網路的傳輸。

✅ E. Peer identity authentication between VPN gateway and customer gateway: 觸發【鐵則 2】與【鐵則 4】，IPSec 透過 IKE 協定互相驗證兩端 Gateway (Peer) 的身分。

✅ F. Data integrity protection across the Internet: 觸發【鐵則 2】，IPSec 提供防竄改的完整性保護。

結論：正確答案是 C, D, E, F。
(你選了 A, B, D, F，下次記得看到 Site-to-Site VPN 就把 End-to-end 殺掉就對了！)

工程師實務補充
在真實世界的架構中，如果你真的被稽核員要求做到 "End-to-End Encryption"，你不能只依賴 AWS Site-to-Site VPN。你必須在你的地端 Server 和 AWS VPC 內的 EC2 之間，使用 TLS (HTTPS) 或 SSH 憑證。VPN 只是保護「走出你家大門到 AWS 大門」這段容易被偷聽的路而已。

如果你在考試中把 Encrypt at Rest（靜態加密） 跟 S3 畫上等號，那你會失掉至少 30% 以上的安全性與資料庫分數。在 AWS 考試中，「Encrypt at Rest」是一個橫向的安全性需求，幾乎出現在所有存儲與資料庫服務中。💡 Encrypt at Rest 的「全域考點」Cheat Sheet當你在題目中看到 "Encrypt data at rest" 或 "Compliance requirement: data must be encrypted before being written to disk" 時，你的腦袋裡應該彈出這張清單：1. EBS (Elastic Block Store) - 磁碟加密關鍵考點： 加密是在 EC2 虛擬機層次完成的（資料在離開 EC2 進入 EBS 之前就加密了）。考試陷阱： * 你不能直接將一個「未加密」的 EBS 改為「已加密」。正確解法： 建立一個該 Volume 的 Snapshot，然後在 Copy Snapshot 的過程中勾選「Encryption」，最後用這個加密的 Snapshot 建立新的 Volume。2. RDS (Relational Database Service) - 資料庫加密關鍵考點： 開啟加密會同時保護：資料庫執行個體、自動備份、Read Replicas、Snapshots。考試陷阱： 跟 EBS 一樣，現有的未加密 RDS 無法直接轉換。必須透過「建立 Snapshot → 複製並加密 Snapshot → 還原」的步驟。3. DynamoDB - NoSQL 加密關鍵考點： 它是預設強制加密的！你不需要手動開啟。考試細節： 考試可能會問你加密是用誰的 Key？你可以選 AWS 擁有的 Key (預設) 或你自管的 KMS Key (CMK)。4. 其他常見服務EFS (Elastic File System)： 在建立時勾選。Redshift (資料倉儲)： 加密整個叢集（Cluster）。Kinesis： 串流資料在寫入碎片 (Shard) 前可以進行靜態加密。🎯 快速判斷表：看到這個服務，請找靜態加密選項服務分類服務名稱靜態加密 (At Rest) 常考點關聯服務 (Key Manager)存儲S3 / EBS / EFS / Glacier複製 Snapshot 或上傳時設定AWS KMS資料庫RDS / Aurora / DynamoDB建立實例時選擇是否加密AWS KMS資料整合Redshift / OpenSearch叢集層級加密AWS KMS秘密管理Secrets Manager自動加密你的密碼/API KeyAWS KMS⚖️ 考試中的「判斷天秤」如果題目問你「如何實現 Encrypt at Rest」，你的刪除法邏輯應該是：先找 AWS KMS： 幾乎 95% 的 At Rest 題目，正確答案都會提到 AWS Key Management Service (KMS)。如果選項提到 CloudHSM，通常是用在「極高合規需求」或「FIPS 140-2 Level 3」。區分 S3 的三種模式：SSE-S3： AWS 全管，最省事（AES-256）。SSE-KMS： 需要稽核軌跡（CloudTrail）或手動管理 Key 權限。SSE-C： 客戶自己管 Key，AWS 只負責加密動作但不存 Key。注意「現有資源」： 如果題目問「現有的」EBS 或 RDS 怎麼加密，「複製 Snapshot」 永遠是標準答案。專家總結：你的新 Cheat Code**"At Rest" = KMS + (S3, EBS, RDS, DynamoDB, ...) **只要記住：除了 S3 之外，EBS 和 RDS 的加密邏輯（Snapshot Copy）在 SAA 考試中出現的機率極高！
