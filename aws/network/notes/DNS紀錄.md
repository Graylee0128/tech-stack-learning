# AWS Route 53 DNS 紀錄完整知識筆記

---

## 第一部分：核心題型分析

### 🚨 CloudFront + Route 53 的陷阱題

**題目情境**：需要為自訂網域指向 CloudFront

**常見選項**：
- C. 使用 **ALIAS Record** 指向 CloudFront
- D. 使用 **Host Record (CNAME)** 指向 CloudFront

**正確答案**：C（ALIAS Record）

---

## 第二部分：DNS 四大天王

### 📋 完整對比

**類比**：DNS 是一本「網路電話簿」，把網址對應到 IP 或其他地址

#### 1️⃣ A Record (Address 紀錄)

| 屬性 | 說明 |
|------|------|
| **功能** | 把「網址」對應到「IPv4 位址」 |
| **例子** | www.example.com → 198.51.100.1 |
| **使用場景** | 固定 IP 的 EC2 或地端伺服器 |
| **根網域適用** | ✅ 可以 |

#### 2️⃣ AAAA Record (Quad-A 紀錄)

| 屬性 | 說明 |
|------|------|
| **功能** | 把「網址」對應到「IPv6 位址」 |
| **使用場景** | 題目提到 IPv6 時直接選它 |
| **根網域適用** | ✅ 可以 |

#### 3️⃣ CNAME (Canonical Name)

| 屬性 | 說明 |
|------|------|
| **功能** | 把「網址」對應到「另一個網址」 |
| **例子** | blog.example.com → blog.hashnode.com |
| **行為** | 類似電話轉接 |
| **根網域適用** | ❌ **絕對不行** ⚠️ |
| **指向目標** | 任何網址（包括非 AWS） |

**CNAME 的致命限制**：
- 不能設在「根網域 (Zone Apex)」（例如 example.com）
- 只能設在「子網域」（例如 www.example.com）
- DNS 協議規定：CNAME 不能與其他紀錄共存

#### 4️⃣ ALIAS Record (別名紀錄 - **AWS 專屬黑科技**)

| 屬性 | 說明 |
|------|------|
| **功能** | 動態追蹤 AWS 託管服務的 IP |
| **外觀** | 看起來像 A Record |
| **內核** | 行為像 CNAME |
| **根網域適用** | ✅ **可以** |
| **指向目標** | 僅限 AWS 資源（ALB、CloudFront、S3） |
| **費用** | 🆓 指向 AWS 資源時免費 |
| **自動更新** | ✅ IP 變動時自動追蹤（60 秒內） |

---

## 第三部分：CNAME vs ALIAS 終極對決

### 🔄 詳細對比表

| 比較項目 | CNAME | ALIAS（AWS 推薦） |
|---------|--------|---------------|
| **指向目標** | 任何網址（含非 AWS） | 僅 AWS 資源 |
| **根網域** | ❌ 不行（考試地雷） | ✅ 可以！|
| **IP 變動反應** | 被動轉接，無感知 | 自動追蹤更新 |
| **DNS 查詢費用** | 💰 按次計費 | 🆓 免費 |
| **與其他紀錄共存** | ❌ 衝突 | ✅ 可以 |
| **跨帳號指向** | ✅ 支援 | ❌ 不支援 |

## 第四部分：為什麼 C (ALIAS) 才是正解？

### 🎯 核心差異分析

#### ❌ D (CNAME) 的兩大致命缺點

1. **根網域限制**
   - CNAME 不能設在根網域 (example.com)
   - 只能設在子網域 (cdn.example.com)
   - 用戶體驗差：必須輸入 www

2. **動態 IP 問題**
   - CloudFront 的 IP 不是固定的，會隨時變動
   - CNAME 只能被動轉接，無法主動追蹤
   - 需要兩次 DNS 查詢，效率低

#### ✅ C (ALIAS) 的黃金優勢

| 優勢 | 說明 |
|------|------|
| **根網域支援** | 可以設在 example.com |
| **自動追蹤** | IP 變動時自動更新 |
| **費用節省** | 免除 DNS 查詢費用 |
| **AWS 最佳實踐** | 官方強烈推薦 |
| **無縫整合** | 與 CloudFront、ALB、S3 完美搭配 |

## 第五部分：考場關鍵字反射動作

### 💡 三大高頻情境

#### 📍 情境一：根網域 + AWS 服務

**題目特徵**：
- 「你買了網域 mycompany.com（沒有 www）」
- 需要指向「ALB 或 CloudFront」

**反射答案**：**ALIAS Record**

**原因**：mycompany.com 是根網域，CNAME 會報錯

---

#### 📍 情境二：跨帳號或非 AWS 資源

**題目特徵**：
- 指向「另一個 AWS 帳號的資源」
- 指向「非 AWS 的外部網址」

**反射答案**：**CNAME**

**原因**：ALIAS 無法跨帳號，只有 CNAME 能做到

---

#### 📍 情境三：成本優化

**題目特徵**：
- 「降低 DNS 查詢費用」(Reduce DNS query cost)
- 「優化成本」

**反射答案**：**ALIAS Record**

**原因**：ALIAS 指向 AWS 資源時免費

---

## 第六部分：CNAME 為什麼不能放 Root？

### 🔐 DNS 協議的鐵律

#### 技術根源

**CNAME 的霸道性**：
```
如果你在一個名字下設 CNAME，它會告訴 DNS：
「從現在起，這個名字的一切都聽我的！」
```

**Root 網域的必要性**：
- 根網域必須擁有自己的 **SOA**（權威紀錄）
- 根網域必須擁有自己的 **NS**（名稱伺服器）

**衝突結果**：
```
CNAME 設在 Root
    ↓
SOA 和 NS 被擠掉
    ↓
全世界 DNS 迷路
    ↓
整個網域癱瘓 ❌
```

---

### 😫 實務上的三大痛點

#### 痛點一：品牌美感與用戶體驗
- 用戶不想打「www」
- 但 CNAME 不能用在根網域
- 被迫讓用戶輸入 www.example.com

#### 痛點二：動態 IP 的噩夢（**最痛**）
- 雲端服務（ALB、CloudFront）沒有固定 IP
- CNAME 不能用在 root（需要 CNAME）
- A Record 需要固定 IP（但沒有）
- **陷入兩難困境**

#### 痛點三：CDN 效能損失
- 被迫只能用 www.example.com 連 CDN
- 用戶輸入 example.com 需要「301 重定向」
- 增加延遲，增加故障點

---

## 第七部分：AWS 的解決方案 - ALIAS 紀錄

### 🚀 「作弊程式」的黑科技

#### 它如何作弊？

**外觀偽裝**：
```
Route 53 ALIAS 紀錄
    ↓
對外看起來：標準 A Record（滿足 DNS 協議）
    ↓
對內行為：像 CNAME 一樣動態追蹤
```

**底層邏輯**：
1. 設定 ALIAS 指向 ALB
2. Route 53 看起來是「A Record」（不會擠掉 SOA/NS）
3. 內部其實是「CNAME」（動態追蹤 IP）
4. ALB IP 變動時，Route 53 在 60 秒內自動更新

#### 完美解決三個問題

| 問題 | CNAME | ALIAS |
|------|--------|--------|
| 根網域支援 | ❌ | ✅ |
| 動態 IP 追蹤 | ❌ | ✅ |
| DNS 費用 | 💰 | 🆓 |

## 第八部分：其他選項為什麼錯？

### ❌ 常見錯誤選項

#### A. S3 Bucket Policy (權限)
- 權限問題會導致 403 Forbidden
- 但通常在 DNS 確認後才檢查
- 不是首要解決方案

#### B. ElastiCache (Redis/Memcached)
- 這是「快取資料庫」，不是 CDN
- 無法存儲圖片和網頁
- 完全牛頭不對馬嘴

>
> 選「ALIAS Record」絕不會錯。
> 它是 AWS 官方推薦的最佳實踐，
> 既支援根網域，又能自動追蹤動態 IP，
