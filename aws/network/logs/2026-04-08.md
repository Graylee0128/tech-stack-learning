## 1. DX / VIF / DXGW 的分層理解

### `DX`

- `DX = physical link`
- 它解的是：
  - 我有沒有一條專線進 AWS

### `VIF`

- `VIF = logical lane`
- 它解的是：
  - 這條 DX 線上的流量要跑哪種用途

### `DXGW`

- `DXGW = logical distribution hub for DX`
- 它解的是：
  - 一條 DX 不用只綁死在單一 VPC
  - 這條 DX 怎麼更彈性地對接多個 AWS 網路目標

### 今天的核心理解

- `DX`、`VIF`、`DXGW` 是三個不同層次
- 它們不是在回答同一個問題
- 容易混，是因為都在同一條 DX 架構裡出現

### 超短口訣

- `DX = 線`
- `VIF = 車道`
- `DXGW = 匯流排`

## 2. 三種 VIF 的用途

### `Private VIF`

- 用來進 VPC 私網模型
- 常見路徑：
  - `Private VIF -> DXGW -> VGW -> VPC`

**重點：**
- 它不是拿來連 AWS public services
- 比較像傳統 hybrid / 少量 VPC 的模型

### `Public VIF`

- 用來走 AWS public services
- 重點不是「上 internet」
- 而是透過 DX 去碰 AWS 公開服務端點

**常見例子：**
- `S3 public endpoint`
- 某些 AWS 公共服務端點

### `Transit VIF`

- 用來進 transit 架構
- 常見路徑：
  - `Transit VIF -> DXGW -> TGW`

**重點：**
- 它不是直接接 TGW
- 它是大規模、多 VPC、多帳號架構的高頻答案

### 三者快速對照

| 類型 | 解的問題 | 常見後端 |
|------|------|------|
| `Private VIF` | 進私網 VPC 模型 | `VGW` / `DXGW -> VGW` |
| `Public VIF` | 去 AWS public services | AWS public endpoints |
| `Transit VIF` | 進 transit hub 架構 | `DXGW -> TGW` |

## 3. VPN Family 快速區分

### `Site-to-Site VPN`

- `網路對網路`
- 地端網路和 AWS 網路之間的加密連線
- 常用於：
  - 快速 hybrid 接入
  - DX 備援

### `Client VPN`

- `人對網路`
- 給終端使用者遠端接入 VPC 內資源

### `VPN CloudHub`

- 不是一個神秘新元件
- 本質上是：
  - 多個 remote sites
  - 各自透過 VPN
  - terminate 在同一個 `VGW`
  - 透過 AWS 做互通

### 今天建立的口訣

- `S2S VPN = 網點對網點`
- `Client VPN = 人連內網`
- `CloudHub = 多站點 via VGW`

## 4. `VGW` vs `TGW`

### `VGW`

- `VGW = one VPC edge`
- 單一 VPC 的邊界模型
- 適合：
  - 單 VPC
  - 傳統 VPN termination
  - 比較簡單的 hybrid

### `TGW`

- `TGW = many VPC hub`
- 適合：
  - 多 VPC
  - 多帳號
  - 多站點
  - 需要 transitive routing
  - centralized routing / segmentation

### 今天建立的判斷

- `單 VPC 接入` -> 先想 `VGW`
- `多 VPC 接入` -> 對 `TGW` 高度敏感

## 5. `association` vs `propagation`

這是今天補到的 TGW 題型核心。

### `association`

- 決定 attachment **查哪張** TGW route table
- 一個 attachment **只能 association 到 1 張**

### `propagation`

- 決定 attachment 的路由 **散到哪些** TGW route table
- 一個 attachment **可以 propagate 到多張**

### 必背口訣

- `Association = 查表`
- `Propagation = 散表`
- `Associate to one, propagate to many`

### 這類題最常怎麼考

- attachment 明明建了但還是不通
- Prod / Dev 要隔離，但都能碰 Shared Services
- on-prem 只能進某些 VPC，不能進其他 VPC

### 今天建立的核心理解

- TGW 題不是只看有沒有 attach 上去
- 真正關鍵通常是：
  - association
  - propagation
  - VPC route table

## 6. Hosted Zone / PHZ / Split-view DNS

### `Hosted Zone`

- 這是一般名詞，不自帶 public / private 屬性
- 題目只寫 `Hosted Zone` 時，要看語境

### `Public Hosted Zone`

- 給 public internet 查詢
- 常見於：
  - `example.com`
  - `www.example.com`
  - internet-facing app

### `Private Hosted Zone`

- 只對關聯的 VPC 可見
- 用來放：
  - internal domain
  - private DNS records
  - VPC 內 service discovery

### `PHZ`

- 幾乎就是 `Private Hosted Zone` 的縮寫
- 考題若直接寫 `PHZ`，通常就是在指 private DNS 範圍

### `Split-view DNS`

- 同一個 domain
- public 查詢看到一套答案
- VPC 內查詢看到另一套答案

**例子：**
- internet 使用者查 `app.example.com` -> public IP / public endpoint
- VPC 內部查 `app.example.com` -> private IP / internal endpoint

### 今天建立的判斷

- `Hosted Zone` 是一般名詞，不要直接腦補成 private
- `PHZ` 幾乎等於 `Private Hosted Zone`
- 題目若寫：
  - internal only
  - private DNS
  - only resolvable from VPC
  - not resolvable from internet

  就要先想到 `Private Hosted Zone`

## 7. 常見例子與易混點

### 例子 1：DX 架構分層

- `DX`：我有沒有線
- `VIF`：這條線跑哪種流量
- `DXGW`：這條流量怎麼更彈性接到 AWS 內部目標

### 例子 2：CloudHub 為什麼說是「同一個 VGW」

- 一個 `VGW` 對一個 `VPC`
- 但可以有多個 remote sites 對同一個 `VGW` 建 VPN
- 這就是 `VPN CloudHub` 的核心模式

### 例子 3：TGW 隔離題

若需求是：
- Prod 可到 Shared
- Dev 可到 Shared
- Prod 不可到 Dev

通常會想到：
- 多張 `TGW route table`
- 控制不同 attachment 的 `association / propagation`

### 今天最大的易混點

- `DXGW` 看起來像 transit 元件，但不是 `TGW`
- `Client VPN` 看起來也是 VPN，但它服務的是人，不是 site
- `association` 和 `propagation` 很像同一件事，但其實一個是「查哪張表」，一個是「誰被哪些表看到」
- `Hosted Zone` 看起來很中性，但題目如果直接寫 `PHZ`，通常已經明確指定 private DNS
