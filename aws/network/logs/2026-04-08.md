## 1. DX / VIF / DXGW 的分層理解

### `DX`

- `DX = physical link`
- 它解的是：
  - 我有沒有一條專線進 AWS

### `VIF`

- `VIF = logical lane`
- 它解的是：
  - 這條 DX 線上的流量要跑哪種用途

### `DXGW`

- `DXGW = logical distribution hub for DX`
- 它解的是：
  - 一條 DX 不用只綁死在單一 VPC
  - 這條 DX 怎麼更彈性地對接多個 AWS 網路目標

### 今天的核心理解

- `DX`、`VIF`、`DXGW` 是三個不同層次
- 它們不是在回答同一個問題
- 容易混，是因為都在同一條 DX 架構裡出現

### 超短口訣

- `DX = 線`
- `VIF = 車道`
- `DXGW = 匯流排`

## 2. 三種 VIF 的用途

### `Private VIF`

- 用來進 VPC 私網模型
- 常見路徑：
  - `Private VIF -> DXGW -> VGW -> VPC`

**重點：**
- 它不是拿來連 AWS public services
- 比較像傳統 hybrid / 少量 VPC 的模型

### `Public VIF`

- 用來走 AWS public services
- 重點不是「上 internet」
- 而是透過 DX 去碰 AWS 公開服務端點

**常見例子：**
- `S3 public endpoint`
- 某些 AWS 公共服務端點

### `Transit VIF`

- 用來進 transit 架構
- 常見路徑：
  - `Transit VIF -> DXGW -> TGW`

**重點：**
- 它不是直接接 TGW
- 它是大規模、多 VPC、多帳號架構的高頻答案

### 三者快速對照

| 類型 | 解的問題 | 常見後端 |
|------|------|------|
| `Private VIF` | 進私網 VPC 模型 | `VGW` / `DXGW -> VGW` |
| `Public VIF` | 去 AWS public services | AWS public endpoints |
| `Transit VIF` | 進 transit hub 架構 | `DXGW -> TGW` |

## 3. VPN Family 快速區分

### `Site-to-Site VPN`

- `網路對網路`
- 地端網路和 AWS 網路之間的加密連線
- 常用於：
  - 快速 hybrid 接入
  - DX 備援

### `Client VPN`

- `人對網路`
- 給終端使用者遠端接入 VPC 內資源

### `VPN CloudHub`

- 不是一個神秘新元件
- 本質上是：
  - 多個 remote sites
  - 各自透過 VPN
  - terminate 在同一個 `VGW`
  - 透過 AWS 做互通

### 今天建立的口訣

- `S2S VPN = 網點對網點`
- `Client VPN = 人連內網`
- `CloudHub = 多站點 via VGW`

## 4. `VGW` vs `TGW`

### `VGW`

- `VGW = one VPC edge`
- 單一 VPC 的邊界模型
- 適合：
  - 單 VPC
  - 傳統 VPN termination
  - 比較簡單的 hybrid

### `TGW`

- `TGW = many VPC hub`
- 適合：
  - 多 VPC
  - 多帳號
  - 多站點
  - 需要 transitive routing
  - centralized routing / segmentation

### 今天建立的判斷

- `單 VPC 接入` -> 先想 `VGW`
- `多 VPC 接入` -> 對 `TGW` 高度敏感

## 5. `association` vs `propagation`

這是今天補到的 TGW 題型核心。

### `association`

- 決定 attachment **查哪張** TGW route table
- 一個 attachment **只能 association 到 1 張**

### `propagation`

- 決定 attachment 的路由 **散到哪些** TGW route table
- 一個 attachment **可以 propagate 到多張**

### 必背口訣

- `Association = 查表`
- `Propagation = 散表`
- `Associate to one, propagate to many`

### 這類題最常怎麼考

- attachment 明明建了但還是不通
- Prod / Dev 要隔離，但都能碰 Shared Services
- on-prem 只能進某些 VPC，不能進其他 VPC

### 今天建立的核心理解

- TGW 題不是只看有沒有 attach 上去
- 真正關鍵通常是：
  - association
  - propagation
  - VPC route table

## 6. Hosted Zone / PHZ / Split-view DNS

### `Hosted Zone`

- 這是一般名詞，不自帶 public / private 屬性
- 題目只寫 `Hosted Zone` 時，要看語境

### `Public Hosted Zone`

- 給 public internet 查詢
- 常見於：
  - `example.com`
  - `www.example.com`
  - internet-facing app

### `Private Hosted Zone`

- 只對關聯的 VPC 可見
- 用來放：
  - internal domain
  - private DNS records
  - VPC 內 service discovery

### `PHZ`

- 幾乎就是 `Private Hosted Zone` 的縮寫
- 考題若直接寫 `PHZ`，通常就是在指 private DNS 範圍

### `Split-view DNS`

- 同一個 domain
- public 查詢看到一套答案
- VPC 內查詢看到另一套答案

**例子：**
- internet 使用者查 `app.example.com` -> public IP / public endpoint
- VPC 內部查 `app.example.com` -> private IP / internal endpoint

### 今天建立的判斷

- `Hosted Zone` 是一般名詞，不要直接腦補成 private
- `PHZ` 幾乎等於 `Private Hosted Zone`
- 題目若寫：
  - internal only
  - private DNS
  - only resolvable from VPC
  - not resolvable from internet

  就要先想到 `Private Hosted Zone`

## 7. 全球入口與流量導向

### Route 53 Routing Policies 快速對照

- `Weighted` = 按比例分流
- `Failover` = 主備切換
- `Latency-based` = 依 AWS latency map 導到最低延遲區域
- `Geolocation` = 依使用者地理位置導流
- `Multivalue` = 多個健康答案，但不是正式 load balancer

### Health Check 核心理解

- `Health Check` 是 DNS failover 題的核心前提
- 題目如果是：
  - primary / secondary
  - active / passive
  - automatic failover

  就要對 `Failover + Health Check` 高度敏感

### `CloudFront`

- `CloudFront = CDN + caching + edge delivery`
- 適合：
  - HTTP / HTTPS
  - 靜態內容
  - 可快取內容
  - WAF / private content / signed URL

### `Global Accelerator`

- `GA = Anycast IP + no caching + TCP/UDP`
- 適合：
  - 固定 Anycast IP
  - non-HTTP 應用
  - fast multi-region failover
  - performance without caching

### `CloudFront` 為什麼是兩段連線

- `Viewer -> CloudFront`
- `CloudFront -> Origin`

也就是說：
- 使用者連到的是 edge
- CloudFront 再自己去連 origin

所以題目若考：
- TLS termination 在哪裡
- CloudFront 到 origin 要不要 HTTPS
- custom domain cert 放哪裡

都不能把它想成單一端到端連線

### `CloudFront` vs `GA`

- `Need caching` -> `CloudFront`
- `Need fixed Anycast IP` -> `Global Accelerator`
- `CloudFront` 不是主打固定 global IP 入口
- 題目若提：
  - allowlist by IP
  - hardcoded IP
  - static Anycast IP

  通常更偏 `GA`

### `ALB / NLB / GWLB`

- `ALB = L7 routing`
- `NLB = static IP / non-HTTP / PrivateLink`
- `GWLB = security appliance scaling`

### 今日建立的高頻口訣

- `比例分流 -> Weighted`
- `主備切換 -> Failover`
- `最低延遲 -> Latency-based`
- `看人在哪 -> Geolocation`
- `Need caching -> CloudFront`
- `Need fixed Anycast IP -> GA`
- `ALB = L7`
- `NLB = static IP`
- `GWLB = inspection`

## 8. CloudFront 安全題

### `Signed URL`

- 用來保護單一物件
- 適合：
  - 單一檔案下載
  - 臨時授權連結
  - 一次性或短時效分享

**口訣：**
- `單一物件 -> Signed URL`

### `Signed Cookie`

- 用來保護一整批內容
- 適合：
  - 會員區
  - 一整組影片 / 文件
  - 同一 session 內反覆讀多個受保護資源

**口訣：**
- `一批內容 -> Signed Cookie`

### `OAC`

- `OAC = Origin Access Control`
- 解的是：
  - 不讓使用者繞過 CloudFront 直接打 S3 origin
  - 只允許 CloudFront 存取 S3 bucket

**口訣：**
- `CloudFront + S3 origin protection -> OAC`

### `OAI`

- 舊版的 CloudFront 對 S3 origin 存取保護方式
- 今天建立的理解是：
  - `OAI = old way`
  - `OAC = current preferred way`

### `WAF`

- 在 CloudFront 題裡解的是：
  - SQLi
  - XSS
  - rate-based abuse
  - 惡意 HTTP / HTTPS request 過濾

**口訣：**
- `Block web attacks -> WAF`

### 今天建立的核心分層

- `Signed URL / Signed Cookie`：控制 viewer 能不能拿內容
- `OAC / OAI`：控制 CloudFront 以外的人能不能碰 origin
- `WAF`：控制惡意 HTTP request 能不能進來

### 今天建立的高頻組合

- 私有 S3 + CloudFront + 單一檔案授權
  - `OAC + Signed URL`
- 私有 S3 + CloudFront + 一整批付費內容
  - `OAC + Signed Cookie`
- CloudFront 前面擋 web attack
  - `WAF`

## 9. VPC 流量路徑總整理

### 核心腦圖

- `VPC = regional boundary`
- `Subnet = AZ boundary`
- `+1 = gateway`
- `+2 = AmazonProvidedDNS`

### 今天真正補進來的解題順序

看到 VPC 流量題，不要先背服務名詞，先按這個順序拆：

1. `誰發起流量`
   - EC2、on-prem、internet 使用者、另一個 VPC、AWS service
2. `流量要去哪裡`
   - internet、S3、另一個 VPC、on-prem、某個 private service
3. `這是路由問題還是存取控制問題`
   - route table / IGW / NAT / endpoint
   - SG / NACL
4. `這題是不是其實在考 private connectivity`
   - Gateway Endpoint
   - Interface Endpoint
   - PrivateLink

### 一句總口訣

- `先看來源和目的，再看 route，最後才看 SG / NACL / DNS`

### `IGW / NAT / EOIGW`

- `IGW`：雙向 internet 入口
- `NAT Gateway`：IPv4 private subnet 主動出網
- `Egress-only IGW`：IPv6 outbound only

#### 三者怎麼在腦中分層

- `IGW`
  - 解的是 public internet connectivity
  - internet 可以主動打進來，但前提是 instance 有 public IP 且規則允許
- `NAT Gateway`
  - 解的是 private subnet 主動出網
  - internet 不能主動對 private instance 建立新連線
- `Egress-only IGW`
  - 解的是 IPv6 版本的 outbound-only internet access
  - 很多題就是在考你不要把 IPv6 也套 NAT 的思路

#### 高頻判斷

- `IPv4 private subnet 要主動更新套件` -> `NAT Gateway`
- `IPv6 only workload 要能出網但不接受外部主動連線` -> `Egress-only IGW`
- `對外網站入口` -> `IGW + public IP + SG`

### Public subnet 成立條件

- route table 有 `0.0.0.0/0 -> IGW`
- instance 有 public IP / EIP

#### 這裡最容易搞混的點

- `public subnet` 和 `public instance` 不是同一件事
- subnet 只代表：
  - 它有沒有對 IGW 的 route
- instance 能不能真的被 internet 打到，還要看：
  - 有沒有 public IP / EIP
  - SG 是否放行
  - NACL 是否沒擋

#### 快速例子

- 同一個 public subnet 裡
  - `EC2-A` 有 public IP -> 可以成為 internet-facing instance
  - `EC2-B` 沒 public IP -> 仍然不能直接被 internet 連到

### `SG / NACL`

- `SG = ENI + stateful + allow only`
- `NACL = subnet + stateless + allow/deny`

#### 真正常考的是什麼

- `SG`
  - 比較像 instance / ENI 身上的門禁
  - 回程流量通常自動放行
- `NACL`
  - 比較像 subnet 邊界的防火牆
  - 回程規則要自己另外放
  - 題目如果故意提 `deny`，通常是在暗示 `NACL`

#### 我今天整理的排查順序

1. 先看 route table 有沒有到正確 target
2. 再看 SG 有沒有放對 inbound / outbound
3. 再看 NACL 有沒有把 request 或 return path 擋掉

#### 一句對照

- `SG 題常像「這台機器能不能進出」`
- `NACL 題常像「這個 subnet 邊界有沒有擋掉」`

### `Flow Logs / Traffic Mirroring`

- `Flow Logs = metadata`
- `Traffic Mirroring = packet payload inspection`

#### 題型差異

- `Flow Logs`
  - 看接受 / 拒絕、來源 / 目的、protocol 等 metadata
  - 不會給你 payload
- `Traffic Mirroring`
  - 拿真正 packet 去做 IDS / IPS / 深度分析

#### 快速判斷

- `想看誰被 ACCEPT / REJECT` -> `Flow Logs`
- `想看封包內容或送去檢測 appliance` -> `Traffic Mirroring`

### `Gateway Endpoint / Interface Endpoint`

- `Gateway Endpoint = S3 / DynamoDB only`
- `Interface Endpoint = ENI + DNS + PrivateLink`

#### 這兩個最常怎麼考

- `Gateway Endpoint`
  - 幾乎就是 `S3 / DynamoDB`
  - 重點是 private access，不經 internet，不經 NAT
  - 靠 route table 生效
- `Interface Endpoint`
  - 支援更多 AWS service 與第三方 SaaS
  - 本質是 `PrivateLink`
  - 靠 ENI + DNS，不是靠 route table 直覺解題

#### 我今天整理的選型句

- `S3 / DynamoDB private access` -> `Gateway Endpoint`
- `其他 AWS service 或 SaaS private access` -> `Interface Endpoint`
- `Provider / consumer / NLB-backed service` -> 往 `PrivateLink` 想

### `VPC Peering`

- `one-to-one`
- `non-transitive`

#### 題目常怎麼陷阱你

- A peering B
- B peering C
- 題目問 A 可不可以經 B 到 C

答案通常是：
- 不行，因為 `Peering = non-transitive`

#### 什麼時候不該再用 peering

- many VPCs
- centralized routing
- hybrid hub
- segmentation

看到這些就該開始改想 `TGW`

### 今天建立的高頻口訣

- `IPv4 private outbound -> NAT GW`
- `IPv6 outbound only -> EOIGW`
- `S3 / DynamoDB private -> Gateway Endpoint`
- `Other private services -> Interface Endpoint`
- `Peering = one-to-one, non-transitive`

### 今天補上的 3 個「連不通」排查模板

#### 模板 1：Internet 打不到 EC2

先查：
1. subnet route 有沒有到 `IGW`
2. instance 有沒有 public IP / EIP
3. SG 有沒有放對 port
4. NACL 回程有沒有放

#### 模板 2：Private subnet EC2 出不了網

先查：
1. private subnet route 有沒有到 `NAT Gateway`
2. NAT Gateway 是否在 public subnet
3. NAT 所在 subnet 是否能經 `IGW` 出網
4. SG / NACL 是否擋掉回程

#### 模板 3：EC2 打 AWS service 沒走 private path

先查：
1. 這題是不是應該用 `Gateway Endpoint` 還是 `Interface Endpoint`
2. route table 或 private DNS 有沒有正確設好
3. endpoint policy / SG 是否有限制

### 今天寫下的流量判斷口訣

- `Internet <-> EC2` 先想 `IGW + public IP`
- `Private EC2 -> Internet` 先想 `NAT Gateway`
- `IPv6 outbound only` 先想 `EOIGW`
- `EC2 -> S3 / DynamoDB private` 先想 `Gateway Endpoint`
- `EC2 -> 其他 private service` 先想 `Interface Endpoint`
- `VPC 對 VPC 少量互聯` 先想 `Peering`
- `VPC 對 VPC 大規模互聯` 先想 `TGW`

## 10. PrivateLink 題型

### PrivateLink 本質

- `PrivateLink = private service exposure`
- 它不是 whole-network connectivity
- 它是把某個服務私下暴露給別人用

### 為什麼它會被單獨考

因為它在解的不是「兩張網怎麼互通」，而是：

- 我要把一個服務提供給很多別人
- 不想做 peering
- 不想暴露到 internet
- CIDR overlap 也不想成問題
- 我只想開某個 service，不想整個 VPC 互通

這種題型非常像 ANS 的典型選型題

### 常見模型

- Provider 端：`NLB`
- Consumer 端：`Interface Endpoint`

### 架構圖

```text
Consumer Account / VPC                              Provider Account / VPC
+----------------------------------+               +----------------------------------+
| Consumer Subnet                  |               | Provider Subnet                  |
|                                  |               |                                  |
| EC2 / App                        |               | App Servers / Internal API       |
|    |                             |               |    ^                             |
|    | DNS resolves service name   |               |    | target group                 |
|    v                             |               |    |                             |
| Interface Endpoint (ENI)         |==============>| NLB                              |
| (PrivateLink endpoint)           |  AWS backbone | (Endpoint Service)               |
|                                  |   private     |                                  |
+----------------------------------+   path        +----------------------------------+

Key ideas:
- Consumer 看到的是自己 VPC 內的一個 Interface Endpoint
- Provider 真正暴露出去的是 NLB 後面的某個 service
- 兩邊不是整個 VPC 打通，而是「某個 service」被私下提供出去
```

#### 用人話串一次

- Provider 先把服務放在 `NLB` 後面
- 對外建立 endpoint service
- Consumer 在自己的 VPC 建 `Interface Endpoint`
- 然後就能私網連到 provider 的服務

### 這題到底在考哪個層次

- `Peering / TGW`
  - 在考 network connectivity
- `PrivateLink`
  - 在考 service connectivity

也就是說：
- `PrivateLink` 不是把兩個 VPC 全打通
- 它是精準地把「某個服務」提供出去

### 什麼時候先想到它

- SaaS provider / consumer
- many customers
- overlapping CIDR
- no peering
- private access to one service only
- cross-account private service connectivity

#### 題幹信號詞

- `overlapping CIDR`
- `provider / consumer`
- `endpoint service`
- `many customer VPCs`
- `private access to application`
- `do not expose entire VPC`

看到這些詞，就要對 `PrivateLink` 高度敏感

### 跟其他服務怎麼分

- `Gateway Endpoint`：幾乎就是 `S3 / DynamoDB`
- `VPC Peering`：整個 VPC 互聯
- `TGW`：network transit / many VPC hub
- `PrivateLink`：service connectivity

### 我今天補的快速比較

| 需求 | 更像哪個答案 |
|------|------|
| 存取 `S3 / DynamoDB` 私網端點 | `Gateway Endpoint` |
| 兩個小型 VPC 要彼此私下互通 | `VPC Peering` |
| 很多 VPC / 帳號要集中式互聯 | `TGW` |
| 只暴露某個 service 給很多 consumer | `PrivateLink` |

### 高頻陷阱

- `PrivateLink` 不是 whole-VPC routing
- `Interface Endpoint` 不只是 endpoint 題，常常骨子裡就是 `PrivateLink`
- 題目如果強調 `CIDR overlap OK`，通常是在暗示不要選 peering
- 題目如果強調 `many customers`，通常是在暗示 provider / consumer 模型

### 這張圖要怎麼看

- `Consumer 端`
  - 會在自己 VPC 裡看到 `Interface Endpoint`
  - 題目如果寫 `endpoint appears as ENI in consumer subnet`，就是在暗示這一側
- `Provider 端`
  - 會把服務掛在 `NLB` 後面
  - 題目如果寫 `endpoint service`、`NLB-backed service`，就是在暗示這一側
- `中間連線`
  - 走 AWS backbone 的 private connectivity
  - 不是 public internet
  - 也不是 VPC Peering 那種整網互通

### 今天建立的口訣

- `PrivateLink = private service exposure`
- `Provider uses NLB`
- `Consumer uses Interface Endpoint`
- `CIDR overlap OK`
- `No peering needed`

## 11. 常見例子與易混點

### 例子 1：DX 架構分層

- `DX`：我有沒有線
- `VIF`：這條線跑哪種流量
- `DXGW`：這條流量怎麼更彈性接到 AWS 內部目標

### 例子 2：CloudHub 為什麼說是「同一個 VGW」

- 一個 `VGW` 對一個 `VPC`
- 但可以有多個 remote sites 對同一個 `VGW` 建 VPN
- 這就是 `VPN CloudHub` 的核心模式

### 例子 3：TGW 隔離題

若需求是：
- Prod 可到 Shared
- Dev 可到 Shared
- Prod 不可到 Dev

通常會想到：
- 多張 `TGW route table`
- 控制不同 attachment 的 `association / propagation`

### 今天最大的易混點

- `DXGW` 看起來像 transit 元件，但不是 `TGW`
- `Client VPN` 看起來也是 VPN，但它服務的是人，不是 site
- `association` 和 `propagation` 很像同一件事，但其實一個是「查哪張表」，一個是「誰被哪些表看到」
- `Hosted Zone` 看起來很中性，但題目如果直接寫 `PHZ`，通常已經明確指定 private DNS
- `CloudFront` 和 `GA` 都在全球入口題出現，但一個主打 caching，一個主打 fixed Anycast IP
- `Multivalue` 看起來像 load balancing，但它不是正式 LB
- `Latency-based` 和 `Geolocation` 很容易混，一個看延遲，一個看位置
- `Signed URL / Cookie` 在保護 viewer access，不是在保護 origin
- `OAC` 在保護 origin，不是在做 viewer access token
- `OAI` 和 `OAC` 解的是同類問題，但新題型更偏 `OAC`
- `Interface Endpoint` 題表面在考 endpoint，骨子裡常在考 `PrivateLink`
- `PrivateLink` 在解 service connectivity，不是在解 whole-network routing
- `Gateway Endpoint` 和 `PrivateLink` 很容易混，但前者幾乎就是 `S3 / DynamoDB` 專用題型
- `public subnet` 不等於裡面的每台 instance 都 public reachable
- `IGW`、`NAT Gateway`、`EOIGW` 都和出網有關，但一個是雙向、一個是 IPv4 outbound、一個是 IPv6 outbound
- `SG` 與 `NACL` 最大差別不是名字，而是 `stateful` 與 `stateless`
- `Gateway Endpoint` 靠 route table，`Interface Endpoint` 更常和 DNS / ENI / PrivateLink 一起想

## 12. Security + 低頻收尾包

### 四個安全元件先分層

- `WAF = L7 web request filtering`
- `Shield = DDoS protection`
- `Network Firewall = VPC / subnet 邊界的 network traffic inspection`
- `DNS Firewall = DNS query control`

### 我今天建立的快速判斷

- `SQLi / XSS / rate-based rules` -> `WAF`
- `volumetric attack / DDoS` -> `Shield`
- `centralized egress inspection / network filtering` -> `Network Firewall`
- `block malicious domains / restrict DNS resolution` -> `DNS Firewall`

### `Firewall Manager`

- 它不是新的封包檢查引擎
- 它是跨帳號 / Organizations 語境下的集中治理工具
- 題目如果提：
  - many accounts
  - centralized policy enforcement
  - consistent security policy

  就要對 `Firewall Manager` 高度敏感

### 這組題最容易混的地方

- `WAF` 是 web layer，不是 DDoS 本體
- `Shield` 是 DDoS，不是 SQLi / XSS 規則引擎
- `DNS Firewall` 管的是 resolver query，不是整體網路流量
- `Network Firewall` 管的是 network path，不是單純 DNS 名稱解析

## 13. 今天新增的低頻辨識骨架

### EKS / 網路治理

- `EKS pod IP` 常和 `VPC CNI` 一起想
- `Security Groups for Pods` 是 pod 級別隔離關鍵字
- `IPAM = CIDR governance`
- `AWS Network Manager = network visibility`

### Hybrid Services

- `Directory Service = AD / identity integration`
- `WorkSpaces = virtual desktop`
- `FSx = managed file system`
- `Storage Gateway = hybrid storage`

### DR 模式最短骨架

- `Backup and Restore` = 最便宜、恢復最慢
- `Pilot Light` = 保留最小核心
- `Warm Standby` = 小規模常駐、可快速放大
- `Active-Active / Multi-site` = 最快、最貴

### Cost 題最常見取捨

- `大量打 AWS services 還走 NAT` -> 應懷疑改成 endpoint
- `不需要大規模 transit 卻上 TGW` -> 應懷疑是否過度設計
- `要求高頻寬穩定主路徑` -> 常偏 `DX`
- `只要快速加密備援` -> 常偏 `VPN`

## 14. Directory Service / WorkSpaces / DR / Cost 展開筆記

### Directory Service 三種常見答案

#### `AWS Managed Microsoft AD`

- AWS 幫你管的 Microsoft AD
- 題目若要：
  - AWS 內有 managed AD
  - 支援比較完整的 Microsoft AD 功能
  - 給 WorkSpaces、FSx for Windows、domain join 等服務使用

  常常是最像考試正解的答案

#### `AD Connector`

- 不是把 AD 搬進 AWS
- 而是讓 AWS 服務去借用你既有的 on-prem AD
- 題目若強調：
  - 沿用既有 corporate AD
  - 不想在 AWS 重建 directory
  - AWS 服務仍要做 AD authentication

  常往這個答案想

#### `Simple AD`

- 較輕量、較舊、較簡化
- 不是考試最主流的 enterprise 答案
- 但如果題目寫的是：
  - 小型組織
  - 輕量 directory
  - 幾百人規模

  還是可能出現

### 今天補到的細節

- `Simple AD Small` 大致對應小規模使用者語境
- 如果題目明確寫到超過幾百人、較完整 AD 能力、企業等級整合，通常會更偏 `Managed Microsoft AD`
- 如果題目要的是「不要重建 AD，只是讓 AWS 服務去用既有 AD」，則更偏 `AD Connector`

### Directory Service 題幹信號詞

- `domain join`
- `Windows authentication`
- `Kerberos`
- `Group Policy`
- `WorkSpaces requires directory`
- `FSx for Windows needs AD`

### 一句判斷

- `AWS 內自己要一套 AD` -> `Managed Microsoft AD`
- `借用既有 on-prem AD` -> `AD Connector`
- `小型 / 輕量` -> 才考慮 `Simple AD`

### `WorkSpaces`

- `WorkSpaces = managed virtual desktop / VDI`
- 它不是 VPN，也不是一般 EC2
- 它解的是：
  - 遠端員工
  - 企業桌面
  - persistent desktop
  - Windows 或 Linux desktop in AWS

### 最常和誰混

- `Client VPN`
  - 只是連線方式，不是桌面
- `AppStream 2.0`
  - 比較像單一應用程式串流，不是完整桌面

### 一句判斷

- `完整桌面` -> `WorkSpaces`
- `單一應用串流` -> `AppStream 2.0`
- `只是要能連進 VPC` -> `Client VPN`

### DR 模式為什麼要分層背

因為考試常常不是直接問名詞，而是問：

- 成本最低是哪個
- 恢復最快是哪個
- 哪個保留最小核心
- 哪個平常就有一套縮小版環境

### `Backup and Restore`

- 平常只保留備份
- 出事時再還原整個環境
- 最便宜
- 通常最慢

### `Pilot Light`

- 平常保留最小核心元件在雲上運行
- 例如：
  - database replication
  - 最小控制平面
  - 關鍵設定與 image
- 出事時還要把 app / web / compute 大量拉起來

### `Warm Standby`

- 平常就有一套縮小版但可工作的完整環境
- 例如：
  - ALB 已經在
  - app tier 已經有少量節點
  - DB 也在跑
- 出事時主要是 `scale up`，而不是從頭 `build up`

### 今天補到的最重要理解

`Warm Standby` 比 `Pilot Light` 快，不是名字比較熱而已，而是：

- `Pilot Light`
  - 還有很多應用層與容量要在災難當下補做
- `Warm Standby`
  - 整套路徑已經存在，只是規模比較小

所以：

- `Pilot Light = 核心活著`
- `Warm Standby = 整套活著，只是縮小版`

### `Active-Active / Multi-site`

- 兩邊都在跑
- 最快
- 最貴
- 複雜度也最高

### DR 一句排序

- `Backup and Restore`
- `Pilot Light`
- `Warm Standby`
- `Active-Active`

這條順序大致就是：
- 成本從低到高
- 恢復速度從慢到快

### Cost 題最常考的 4 個方向

#### 1. NAT 成本

- 如果大量流量其實只是去 AWS service
- 卻還一直經 NAT
- 就要先懷疑可否改成：
  - `Gateway Endpoint`
  - `Interface Endpoint`

#### 2. TGW 是否過重

- 如果只是少量 VPC 互聯
- 或只是暴露單一 service
- 卻用了 transit 方案
- 題目常在暗示設計過重、成本偏高

#### 3. DX vs VPN

- `DX`
  - 穩定、正式、高頻寬、低延遲
  - 成本通常較高
- `VPN`
  - 便宜、快部署
  - 常當備援或過渡方案

#### 4. DR 成本

- 越快恢復，通常越貴
- 題目如果同時要：
  - 極低 RTO / RPO
  - 又要最低成本

  通常就是在逼你做 tradeoff

### 今天的壓縮口訣

- `Managed Microsoft AD = AWS 內一套 AD`
- `AD Connector = 借用既有 AD`
- `WorkSpaces = 雲端桌面`
- `Warm Standby 比 Pilot Light 快，因為前者是在 scale up，不是在 build up`
- `NAT 成本高，先懷疑 endpoint`
- `需求小，不要亂上 TGW`
