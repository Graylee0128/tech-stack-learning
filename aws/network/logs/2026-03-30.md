## 今日學習主題

- `VGW`：單一 VPC 接地端的傳統邊界閘道
- `TGW`：多 VPC / 多站點的 hub
- `DXGW`：讓 DX 連接跨 Region / 多 VPC 的邏輯中介
- `LOCAL_PREF`：主要控制 outbound path
- `AS_PATH prepend`：常用來影響 inbound path
- `MED`：鄰居偏好提示，不是萬能控制鈕
- AWS 路由優先序：`Longest Prefix -> Static -> DX -> VPN static -> VPN BGP`

## 今日關鍵記憶點

- `VGW = 單 VPC 邊界`
- `TGW = 多網路中樞`
- `DXGW = DX 的全球擴展器`
- `DX` 預設優先於 `VPN`
- `DXGW` 不提供 VPC-to-VPC transitive routing
- `Longest prefix` 比 BGP 屬性更早決定選路
- 想讓特定網段改走另一條路，先想更具體 prefix

## 今日自測

- 單一 VPC 要接地端 VPN：`VGW`
- 很多 VPC 要互通：`TGW`
- 同 prefix 且無其他條件干擾時：`DX` 優先
- `DXGW` 不拿來做 VPC-to-VPC transit
- `LOCAL_PREF` 主要控制出站偏好
- 影響對方回流路徑常先想 `AS_PATH prepend`
- DX `172.16.0.0/16` vs VPN `172.16.10.0/24` 時，`/24` 先贏
- 手動 `static route` 會壓過 propagated DX route

## 今日題型演練

- 題型 1：特定 CIDR 要改走 VPN
  - 正解：在 VPN 端宣告更具體的 prefix
  - 判斷依據：`Longest prefix` 先於 `DX vs VPN`
- 題型 2：大量 VPC、多帳號、多 Region、單組 DX 要擴展
  - 正解：`Transit VIF -> DXGW -> TGW`
  - 判斷依據：要 scalable hub 架構，不做 per-VPC 直連
- 題型 3：兩個 VPC 掛同一個 DXGW 是否互通
  - 正解：不可以
  - 判斷依據：`DXGW` 不提供 VPC-to-VPC transitive routing

## 錯題 / 卡點

- 目前無正式錯題
- 容易混淆點：`DXGW` 很像 transit 元件，但它不是 `TGW`
- 容易混淆點：想改某個特定網段走向時，要先看 `Longest prefix`，不要先跳去想 BGP 屬性
