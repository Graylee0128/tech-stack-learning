# 為什麼小團隊選路徑、大團隊選子域？

> **前言**：這種選擇背後有個管理學原則（Melvin E. Conway 在 1967 年提出過類似觀察）：**組織結構往往會映射到技術架構上**。所以架構選擇不只是技術問題，還涉及組織怎麼協作。

---

## 快速結論

| 團隊規模 | 推薦方案 | 理由 |
|---------|---------|------|
| **< 20 人** | **路徑**<br/>`app.example.com/api` | 一個團隊、一個代碼倉庫、一個部署 |
| **20-50 人** | **混合**<br/>子域分服務 + 路徑分版本 | 前後端開始分工，但還是統一發布 |
| **> 50 人** | **子域**<br/>`api.example.com`、`auth.example.com` | 多個獨立服務團隊，各自部署 |

---

## 小團隊為什麼選路徑？

### 場景：3-10 人初創

```
組織結構：
├─ 扁平，沒有明確分工
├─ 大家都是全棧
├─ 一起開會、一起決策
└─ 一週可能發布多次

代碼結構：
├─ 一個倉庫（或一個主分支）
├─ 前後端代碼在一起
└─ 一次 commit 可能改多個功能

推薦架構：路徑
└─ app.example.com/api
└─ app.example.com/dashboard
└─ app.example.com/admin
```

### 為什麼路徑最高效？

1. **協調成本最低**
   - 不用決策「誰管哪個子域」
   - 沒有跨團隊邊界
   - 溝通簡單

2. **部署最簡單**
   - 一個 LB、一個應用實例
   - 一個 SSL 證書
   - 一個部署流程

3. **快速反覆運算**
   - 想改什麼就改，一起發布
   - 不用等其他團隊
   - 最適合創業期快速迭代

### 真實例子

Stripe（初期）、Airbnb（初期）都是這樣：
```
stripe.com/api     ← 支付 API
stripe.com/dashboard  ← 後台
stripe.com/docs    ← 文檔

一個團隊，一份代碼，一次發布
```

## 中等團隊為什麼開始用子域？

### 場景：20-50 人，開始分工

```
組織結構：
├─ 前端團隊（5-10 人）
├─ 後端團隊（8-15 人）
├─ 運維團隊（2-3 人）
└─ 開始有明確的團隊邊界

代碼結構：
├─ 前端倉庫
├─ 後端倉庫
└─ 分別部署

推薦架構：混合（子域 + 路徑）
├─ app.example.com（前端）
├─ api.example.com（後端）
└─ api.example.com/v1、/v2（版本）
```

### 為什麼需要子域分工？

1. **團隊邊界變清晰**
   - 前端團隊負責 `app.example.com`
   - 後端團隊負責 `api.example.com`
   - CORS 變成必須要配置的問題

2. **獨立部署的需求**
   - 前端想每天發布多次
   - 後端想每週穩定發布一次
   - 子域允許這樣的靈活性

3. **不同的技術棧**
   - 前端：React/Vue + Node.js
   - 後端：Go/Python + 數據庫
   - 各自用不同的基礎設施

### 成本權衡

```
成本：
├─ 多一個子域（DNS）→ 幾乎沒有成本
├─ 可能多一個 LB → +$15-30/月
└─ 總成本：小

收益：
├─ 前後端團隊獨立開發
├─ 獨立部署、獨立縮放
└─ 減少協調成本 → 價值很高
```

### 真實例子

Notion（中期）、Slack 都是這樣：
```
app.slack.com   ← 前端
api.slack.com   ← 後端
```

## 大團隊為什麼一定要用子域？

### 場景：50+ 人，多個獨立服務團隊

```
組織結構：
├─ API Gateway Team（5 人）
├─ Auth Service Team（5 人）
├─ Payment Service Team（5 人）
├─ Recommendation Team（8 人）
├─ Infrastructure Team（5 人）
└─ 每個團隊有完整的 CI/CD、監控、告警

代碼結構：
├─ repo/api-gateway/
├─ repo/auth-service/
├─ repo/payment-service/
├─ repo/recommendation-service/
└─ 各自獨立開發、各自獨立部署

推薦架構：子域（微服務）
├─ api.example.com → API Gateway Team
├─ auth.example.com → Auth Service Team
├─ payment.example.com → Payment Service Team
├─ recommend.example.com → Recommendation Team
└─ 清晰的所有權（Service Owner）
```

### 為什麼一定要子域？

1. **團隊太多，無法協調**
   - 50+ 人全體開會？不現實
   - 必須分出明確的所有權
   - 子域 = 團隊的「責任邊界」

2. **部署節奏完全獨立**
   - Auth Team 每天發布 3 次
   - API Team 每週發布 1 次
   - Payment Team 每月審慎發布 1 次
   - 各自走各自的 CI/CD

3. **故障隔離**
   - Auth 服務故障 → 只影響登錄
   - API 服務故障 → 只影響數據
   - 不會全體崩潰

4. **獨立縮放**
   - Payment 黑五流量暴增 → 只縮放 payment 服務
   - 不用縮放整個應用

### 成本變化

```
成本會上升：
├─ 更多子域
├─ 多個 LB
├─ Kubernetes 運維成本
├─ 每個服務的監控、告警
└─ 總成本：高

但收益也高：
├─ 團隊完全獨立
├─ 不互相阻塞
├─ 故障隔離
└─ 可以獨立縮放
└─ 快速迭代能力 ✓
```

### 真實例子

Netflix、Uber、Amazon 都是這樣：
```
api.netflix.com
auth.netflix.com
payment.netflix.com
recommendation.netflix.com
...
每個服務一個團隊，完全獨立
```

## 開發模式如何影響選擇？

### 集中式開發（Monolith in Git）

```
一個倉庫，一個應用
└─ src/
   ├─ frontend/
   ├─ backend/
   ├─ admin/
   └─ 一起提交、一起部署

推薦：路徑
理由：代碼改動牽一髮動全身，必須一起測試一起發布
```

### 分散式開發（Microservices in Separate Repos）

```
多個獨立倉庫
├─ repo/api/
├─ repo/auth/
├─ repo/payment/
└─ 各自 merge、各自部署

推薦：子域
理由：服務獨立，DNS 邊界清晰表達所有權
```

### 混合式開發（Monorepo）

```
一個倉庫，多個服務（Google、Facebook 風格）
├─ packages/api/
├─ packages/auth/
├─ packages/payment/
└─ 共享依賴，各自部署

推薦：子域（邏輯清晰）
理由：代碼一起管理，但服務可獨立部署
      子域反映了這種邏輯結構
```

## 上線節奏如何影響選擇？

### 快速迭代（每天發布多次）

```
特點：代碼改動頻繁，風險高

風險：
└─ 單體應用一個地方改壞 → 全體受影響

解決：用子域分解風險
├─ api.example.com（穩定，改動少）
├─ ui.example.com（變動快，每小時發布）
└─ 前端故障不影響後端
```

**案例**：Slack 的前端可以每小時更新，後端保持穩定

### 穩定發布（每季度一次）

```
特點：長期計劃、謹慎

推薦：路徑簡化架構
理由：
├─ 協調簡單
├─ 一次全發
├─ 完整測試
└─ 成本低
```

**案例**：企業軟件（SAP、Oracle）每次發布都是大事件

## 實務決策樹

```
START：現在要決定用什麼？

Q1：團隊多少人？
├─ < 20 人 → Q2
├─ 20-50 人 → Q3
└─ > 50 人 → 用子域 ✓

Q2（小團隊）：代碼分散嗎？
├─ 是（已有多個 repo）→ 用子域 ✓
└─ 否（一個 monolith）→ 用路徑 ✓

Q3（中等團隊）：前後端分工很清晰嗎？
├─ 是，經常衝突 → 用子域 ✓
└─ 否，還是一起發布 → 用混合 ✓

Q4（大團隊）：已經微服務化嗎？
├─ 是 → 用子域 ✓
└─ 否 → 考慮開始遷移 ⚠️
```

## 成長路線圖

```
時間軸：

創業初期          小有成就          規模化           企業級
(< 10 人)       (10-50 人)      (> 50 人)       (> 500 人)
   ↓               ↓               ↓               ↓
  路徑      →  路徑+子域  →  子域+微服務  →  完全微服務化

  簡單高效   →   權衡中期   →   複雜但強大   →  企業級協作
```

### 從路徑升級到子域的信號

✅ **該升級了**：
- 前後端團隊開始常常衝突
- 想要獨立發布
- 開始有多個獨立的倉庫
- 部署變得複雜（大家等著一起發）

❌ **還不用升級**：
- 團隊還是一起開會討論
- 代碼改動涉及前後端都要改
- 還沒有明確的團隊邊界

## 實踐建議

### 不要過度工程化

❌ **常見錯誤**：
```
3 人初創用微服務架構
  ├─ Kubernetes
  ├─ 10 個獨立服務
  ├─ 複雜的 CI/CD
  └─ 每個人都得學運維 → 死掉了
```

✅ **應該怎麼做**：
```
3 人初創用單體 + 路徑
  ├─ 一個 app.example.com/api
  ├─ 一個倉庫、一個部署
  └─ 快速迭代、快速成長
```

---

### 隨著成長調整，不用內疚

**常見的進化**：
```
Year 1：路徑 ✓
        app.example.com/api、/dashboard

Year 2：混合開始出現
        app.example.com（前端）
        api.example.com（後端）
        有些團隊想獨立部署

Year 3：微服務化加速
        api.example.com
        auth.example.com
        payment.example.com
        完全獨立的團隊結構
```

這種演進是**完全正常的**，甚至是必然的。

---

## 總結

你的架構選擇應該反映：
1. **團隊規模** → 越大越需要子域
2. **代碼結構** → Monolith 用路徑、Microservices 用子域
3. **部署節奏** → 快速迭代需要子域分解風險
4. **成本考慮** → 初期成本優化，成長期功能優化

**最簡單的原則**：
- 小團隊 = 路徑（簡單高效）
- 大團隊 = 子域（清晰所有權）
- 中間過渡 = 混合方案（邊界明確）
