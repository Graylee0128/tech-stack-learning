# 網路安全嗎？從 HTTP 到 Zero Trust 的演進 {HTTP~ZTN}

> HTTP → HTTPS → mTLS → TLS Termination → NLB/ALB → Service Mesh → Zero Trust (ZTN)
> 其實已經形成一條「架構知識鏈」

## 前言：為什麼會有這些技術？（背景脈絡）

### HTTP 時代的黑暗（1990s 早期）

最初的 **HTTP（Hyper Text Transfer Protocol）** 設計非常簡單：
- ❌ 完全明文傳輸
- ❌ 無加密、無身份驗證
- ❌ 任何中間節點都可以竊聽和篡改

**問題場景**：
```
用戶 ─HTTP明文─→ ISP 可以看 ─→ 網站伺服器
                  ↓
             攻擊者可以：
             • 看你輸入什麼密碼
             • 改變網頁內容
             • 冒充網站身份
```

**當時的解決方案**：
- 相對封閉網絡
- VPN/專線
- 相信 ISP（不現實）

**痛點**：
- 電商無法上線（無法安心輸入信用卡）
- 銀行業無法使用
- 用戶隱私毫無保障

### HTTPS 時代的突破（1990s 末～2000s）

TLS/SSL 引入了 **加密**：

**握手模型**：
```
Client → 驗證 Server 憑證（確認是真正的銀行網站，不是釣魚）
Server → 建立加密通道
            ↓
        用戶輸入的密碼被加密
        傳輸過程中看不到明文
```

**解決的問題**：
- ✅ 加密傳輸（中間節點看不到）
- ✅ Server 身份驗證（確認連到真網站）

**但仍有侷限**：
- ❌ Server 不知道 Client 的真實身份（只靠帳號密碼）
- ❌ 單向驗證（用戶信任銀行，但銀行不驗證用戶）

**企業內網的假設**：
- 當時的企業應用：「防火牆內都是自己人」
- 相信內網邊界（Perimeter Security）
- 不需要強身份驗證

### HTTPS 時代的侷限（1990s~2000s）

最初的 **HTTPS（TLS）** 只做單向驗證：
- ✅ Client 驗證 Server：「確認網站是真的」
- ❌ Server 不驗證 Client：「只靠帳號密碼識別使用者」

這對**網站**夠用，但**企業內網**的成百上千個服務呢？
- 當時的解決方案：**相信內網（信任邊界）**
- 假設：防火牆內的流量都是安全的

### 微服務時代的危機 {HTTPS→mTLS}（2010s 中期）

微服務架構打破了傳統邊界：
- 服務數量爆炸（從幾個變成幾百個）
- 服務間東西向流量（East-West）大增
- **內網邊界消失**：流量不再集中在南北向出入口

核心問題浮現：
- Service A 可以隨意呼叫 Service B？無法驗證服務身份
- 單一入侵者可能橫向移動到所有服務
- 傳統防火牆 + 內網信任模型破產

### Cloud Native + Kubernetes 的衝擊 {mTLS→ZTN}（2015 左右）

容器與 Kubernetes 進一步加劇問題：
- IP 地址動態分配：防火牆規則基於 IP 無法維護
- 服務不停遷移：難以追蹤誰是誰
- **需要 Workload Identity（服務身份）**：而不是 IP 或帳號

這時 **Google BeyondCorp**（2014）提出關鍵概念：
> 「不要信任內網，每次都驗證」→ **Zero Trust**

### NIST 正式定義 Zero Trust（2019）

美國國家標準局（NIST）發佈 **SP 800-207**，定義三大支柱：
1. **強身份驗證**：mTLS、IAM、PKI
2. **持續驗證**：每次連線都重新檢查
3. **最小權限**：只能訪問必要資源

**技術棧的完整演進：**

```
HTTP（明文）
   ↓ 引入加密
HTTPS（Server 認證）{HTTP→HTTPS}
   ↓ 引入雙向驗證
mTLS（客戶端身份驗證）{HTTPS→mTLS}
   ↓ 引入自動化
Service Mesh mTLS（Workload Identity）{mTLS→ServiceMesh}
   ↓ 引入策略層
Zero Trust 架構（認證 + 授權 + 驗證 + 分段）{ServiceMesh→ZTN}
   ↓ 完整模型
```

### 這篇文章的定位

本文深入講解 **HTTP 如何演進到 Zero Trust 架構** 的技術細節與邏輯鏈，適合：
- 學習 AWS 認證考試（ANS）
- 設計微服務網路安全
- 準備面試（架構師視角）
- 理解安全技術的歷史脈絡與必然性

## 一、核心總覽：HTTP→ZTN 知識地圖（完整架構鏈）

```
HTTP（明文傳輸）
   ↓
HTTPS / TLS（Server 認證 + 加密） {HTTP→HTTPS}
   ↓
mTLS（雙向身份驗證） {HTTPS→mTLS}
   ↓
TLS Termination / Passthrough（安全邊界設計） {mTLS→ALB/NLB}
   ↓
Service Mesh mTLS（自動化 Workload Identity） {ServiceMesh→AutomatedmTLS}
   ↓
Zero Trust Architecture（完整安全模型） {ServiceMesh→ZTN}
```

**關鍵理解：**

> mTLS 是 Zero Trust 的「身份與加密基礎層」，不是完整 ZTN 本身。

## 二、TLS vs mTLS：本質差異 {HTTPS→mTLS}（基礎但關鍵）

### 1. 一般 TLS（HTTPS）{HTTP→HTTPS}

**握手模型：**

```
Client → 驗證 Server 憑證
Server → 不驗證 Client 身份（只知道連線）
```

**特點：**

- 單向驗證
- 主要用途：加密 + Server 身份可信
- 常見於：網站、API、CloudFront、ALB

**核心限制：**

- 無法強身份識別 Client（只靠Token/登入）

### 2. mTLS（Mutual TLS）{HTTPS→mTLS}

**握手模型：**

```
Client ⇄ Server
雙方都出示憑證並驗證
```

**流程（簡化）：**

- Client 驗證 Server certificate
- Server 要求 Client certificate
- Server 驗證 Client 憑證（PKI / CA）
- 才建立加密通道

**本質提升：**

| 面向 | TLS | mTLS |
|------|-----|------|
| 加密 | ✔ | ✔ |
| Server 身份驗證 | ✔ | ✔ |
| Client 強身份驗證 | ❌ | ✔ |
| 適合 Zero Trust | ❌ | ✔ |

## 三、TLS Termination 是什麼？{mTLS→ALB/NLB}（ANS + 架構超高頻）

### 定義（最精準）

> TLS Termination = 在某個中間節點（如 ALB）解密 TLS，再用新連線轉發到後端服務。

### 1. ALB TLS Termination 架構 {邊界加密}

**ALB 如何終止 TLS**：

```
Client (HTTPS)
   ↓
ALB（解密TLS）
   ↓
HTTP / HTTPS（新連線）
   ↓
Backend
```

**安全特性：**

- 加密只保護到 ALB
- ALB 可讀取 HTTP Header / Cookie
- 支援 WAF / L7 routing

**適合：**

- Web應用
- CloudFront + ALB
- 一般認證系統

### 2. NLB TLS Passthrough（端到端）{端到端加密}

```
Client (HTTPS)
   ↓（加密保持）
NLB（L4轉發，不解密）
   ↓
Backend（在此完成TLS/mTLS握手）
```

**關鍵特性：**

- 真正 End-to-End TLS
- LB 看不到明文
- 完整支援 mTLS 握手控制

## 四、mTLS 與 ALB / NLB 的架構選型 {mTLS→ALB/NLB}（考試+實務）

### 1. mTLS「應用層驗證」模式 {ALB模式}（最常見）

```
Client → ALB（TLS終止）
        → App / Envoy / Nginx 驗證 Client Cert
```

**特點：**

- ALB 仍可使用
- 應用程式處理憑證邏輯
- 雲原生常見（微服務）

### 2. mTLS「完整握手控制」模式 {NLB模式}（高安全）

```
Client → NLB（Passthrough）
        → Backend 完整 TLS + mTLS 握手
```

**適用場景：**

- 金融系統
- 高合規（PCI / 銀行）
- 自建 PKI
- Zero Trust 高等級環境

**架構本質差異：**

| 項目 | ALB模式 | NLB模式 |
|------|--------|--------|
| TLS終止位置 | Load Balancer | Backend |
| 憑證控制權 | 平台層 | 應用/後端 |
| 安全邊界 | 邊界加密 | 端到端加密 |

## 五、Service Mesh mTLS {ServiceMesh→mTLS}（雲原生核心）

### 定義

> Service Mesh mTLS = 在服務與服務（East-West traffic）之間**自動化**建立雙向憑證驗證與加密通道

**典型架構：**

```
Service A ↔ Sidecar Proxy (Envoy)
           ⇄ mTLS ⇄
Service B ↔ Sidecar Proxy
```

**核心能力：**

- Workload Identity（服務身份）
- 自動憑證輪替
- 東西向流量加密
- 不依賴內網信任

**常見工具：**

- Istio
- Linkerd
- Consul Mesh

## 六、mTLS 與 Zero Trust（ZTN）的真正關係 {ServiceMesh→ZTN}

### 關鍵結論（架構師級）

> mTLS ≠ Zero Trust
> mTLS = Zero Trust 的 **Identity + Encryption 基礎層**（必要但不充分）

### Zero Trust 三大核心支柱（NIST模型）

#### 1️⃣ 強身份（Identity）

**技術：**

- IAM
- PKI
- mTLS
- OIDC / SAML
- Workload Identity（Service Mesh）

#### 2️⃣ 持續驗證（Continuous Verification）

**特徵：**

- 每次連線重新驗證
- 不信任內網
- Context-aware access

**mTLS貢獻：**

✔ 每次連線都重新驗證憑證（符合 ZT 原則：Never trust, always verify）

#### 3️⃣ 最小權限（Least Privilege）

**技術：**

- RBAC / ABAC
- Policy Engine（OPA）
- IAM Policy
- Micro-segmentation

**關鍵點：**

- mTLS 只解決「你是誰」
- 不解決「你能做什麼」

## 七、為什麼 Service Mesh mTLS ≠ 完整 Zero Trust {ServiceMesh→ZTN}

即使系統：

- 全部 mTLS
- 全部加密

仍然可能：

- Service A 可以呼叫所有 Service（無授權限制）

這違反 Least Privilege（最小權限）原則。

因此成熟 ZTN 還需要：

- Authorization Policy
- Policy as Code（PaC）
- Fine-grained Access Control

## 八、Zero Trust 成熟度模型 {HTTP~ZTN}（實務最重要）

> 基於 NIST + CISA 的演進過程

| 等級 | 特徵 | 是否算 ZTN |
|------|------|----------|
| Level 0 | VPN + Firewall + 內網信任 | ❌ |
| Level 1 | IAM + MFA | ❌ 初步 |
| Level 2 | mTLS + 微分段 + 身份導向 | ⭕ 進階 |
| Level 3 | 持續驗證 + PaC + Context-aware | ✔ 成熟ZTN |

## 九、ZTN ≠ 單一技術 {完整架構}（最常見誤解）

**❌ 錯誤公式：**

```
ZTN = mTLS + IAM + PKI（不完整）
```

**✅ 更完整的架構公式：**

```
Zero Trust Architecture =
  Identity (IAM + PKI + mTLS)
  + Encryption (End-to-End TLS)
  + Authorization (RBAC / ABAC / Policy Engine)
  + Continuous Verification
  + Micro-segmentation
  + Observability
```

## 十、Policy as Code（PaC）在 ZTN 的定位 {Authorization→PaC}

**精準定義：**

> PaC = Zero Trust 的「授權與策略自動執行層」

不是 ZTN 本身，而是：

- 落實最小權限
- 自動化安全治理
- 持續策略檢查

**常見工具：**

- OPA (Open Policy Agent)
- Kyverno
- AWS IAM Policy
- Terraform Sentinel

## 十一、考試（AWS ANS）與實務的差異提醒

| 情境 | 考試思維 | 實務思維 |
|------|---------|--------|
| mTLS / Client Certificate | 傾向 NLB | 視TLS終止位置而定 |
| CloudFront + Web | ALB | 仍多用 ALB |
| 高安全合規 | NLB + End-to-End TLS | 金融/大型企業 |

## 十二、最終總結 {HTTP~ZTN}（可當筆記首頁）

### 演進的邏輯鏈

```
HTTP（明文） → HTTPS（Server認證） → mTLS（雙向認證）
  → ServiceMesh自動化 → Zero Trust完整模型
```

### 核心認知

**mTLS** 是一種在 TLS 握手層進行雙向身份驗證的機制，能提供強身份與持續驗證能力，是 Zero Trust Architecture 的 **核心基礎之一**。

**然而**，完整的 Zero Trust 並不只依賴 mTLS，還需要結合：
- **身份層**：IAM、PKI、Workload Identity
- **加密層**：端到端 TLS/mTLS
- **授權層**：RBAC/ABAC、Policy as Code
- **分段層**：微分段、網路策略
- **驗證層**：持續驗證、Context-aware access
- **可觀測層**：Observability、日誌與監控

才能達到真正的**零信任安全模型**。
