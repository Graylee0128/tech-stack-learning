# 狀態 / 行動 / 評估函數 建模筆記

> 橋的子筆記之三，**也是最重要的一份**。
> 能把問題拆成 (state, action, cost/heuristic)，是判斷「準備好走進 heuristic」的 litmus test。

## 為什麼這份筆記是 litmus test

heuristic 演算法的整個語言系統都建立在「問題 = state space + action + evaluation」上：

- A* 需要 heuristic function h(state)
- Local Search 需要 neighbor(state) 與 cost(state)
- Genetic Algorithm 需要 chromosome representation 與 fitness function
- Simulated Annealing 需要 energy function 與 transition

**不會建模 = 連 heuristic 的題目都讀不懂**，更別說選哪個方法。

## 建模四元組

對任何問題，先問自己：

1. **State 怎麼表示？**（資料結構、編碼方式、空間大小）
2. **Action 是什麼？**（從一個 state 能跳到哪些 state？分支度多大？）
3. **Cost / Reward 怎麼算？**（轉移代價、目標函數）
4. **Goal / Termination 條件？**（什麼時候停？最佳化什麼？）

## 範例題（待逐題填）

### 例 1：8-puzzle / 15-puzzle

- State：3×3 棋盤的排列
- Action：空格上下左右移動
- Cost：每步 = 1
- Goal：排列 = 目標排列

### 例 2：TSP（Travelling Salesman）

- State：當前路徑（已訪問城市序列）
- Action：選下一個未訪問城市
- Cost：累計距離
- Goal：訪問所有城市並回到起點

### 例 3：最短路（Dijkstra 視角）

- State：當前節點 + 已知最短距離
- Action：鬆弛相鄰邊
- Cost：邊權
- Goal：到達目標節點

### 例 4：N-Queens（Backtracking 視角）

- State：已放好的皇后位置集合
- Action：在下一列放一個合法位置
- Cost：(N/A，純可行性)
- Goal：放滿 N 個

## 同一個問題，不同建模

同一個問題用不同 state 定義，會導出完全不同的演算法：

- TSP 用「當前路徑」當 state → backtracking / branch and bound
- TSP 用「已訪問集合 + 當前位置」當 state → DP（Held-Karp，O(n²·2ⁿ)）
- TSP 用「整條路徑」當 state → 適合 GA / SA 等 heuristic

**會切換建模 = 真的懂 DSA**。
