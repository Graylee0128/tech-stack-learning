# 狀態 / 行動 / 評估函數 建模筆記

> 橋的子筆記之三，**也是最重要的一份**。
> 能把問題拆成 `state + action + objective + constraints`，是判斷「準備好走進 heuristic / solver」的 litmus test。

## 一句話定義

建模就是把一個模糊的工程問題，壓成可以被演算法、heuristic 或 solver 處理的形式：

```text
state        目前長什麼樣
action       下一步可以怎麼改
objective    我想讓什麼變小或變大
constraints  哪些條件絕對不能違反
```

如果這四件事說不清楚，後面不管用 A*、DP、2-opt、SA、CP-SAT、OR-Tools，都是在賭。

## 為什麼這份筆記是 litmus test

heuristic 演算法的整個語言系統都建立在「問題 = state space + action + evaluation」上：

- A* 需要 `h(state)`，也就是從目前狀態到目標的估計成本。
- Local Search 需要 `neighbor(state)` 與 `cost(state)`。
- Genetic Algorithm 需要 chromosome representation 與 fitness function。
- Simulated Annealing 需要 energy function 與 transition rule。
- OR-Tools / CP-SAT 需要 decision variables、constraints 與 objective。

**不會建模 = 連 heuristic 的題目都讀不懂**，更別說選哪個方法。

這份筆記不是要背一個格式，而是建立一個反射動作：拿到任何題目時，先問「狀態是什麼、動作是什麼、目標是什麼、限制是什麼」。先建模，再選算法。

## 建模四元組

對任何問題，先問自己：

1. **State 怎麼表示？**  
   資料結構、編碼方式、是否可 hash、空間大小、是否能判重。

2. **Action 是什麼？**  
   從一個 state 能跳到哪些 state、分支度多大、是否可逆、是否會破壞 constraints。

3. **Objective / Cost / Reward 怎麼算？**  
   是最小化距離、最大化收益、最小化違規懲罰，還是純可行性問題。

4. **Constraints / Goal / Termination 條件？**  
   什麼一定要滿足、什麼時候停、是找到任一可行解、最短路、最佳解，還是 time limit 內的好解。

實務上建模時可以再補兩個欄位：

5. **Baseline 是什麼？**  
   小樣本 exact、greedy、random、人工規則。沒有 baseline，就無法判斷 heuristic 有沒有意義。

6. **Guarantee 是什麼？**  
   保證最佳、保證近似比、只保證可行、還是沒有保證但可用實驗驗證。

## 最小建模模板

之後每碰到一個新題型，都可以先貼這段再填：

```text
Problem:

State:

Initial state:

Action / Neighbor:

Transition cost:

Objective:

Constraints:

Goal / Termination:

State-space size:

Baseline:

Candidate algorithms:

Optimality guarantee:

Failure risks:
```

填完這張表，通常就已經知道問題比較像 graph search、DP、backtracking、local search、approximation 還是 solver。

## 同一個問題，不同建模

同一個問題用不同 state 定義，會導出完全不同的演算法。

以 TSP 為例：

| State 定義 | Action | 對應方法 | 意義 |
|---|---|---|---|
| 當前路徑 prefix | 選下一個未訪問城市 | Backtracking / Branch and Bound | 精確搜尋，適合小 n 或剪枝強 |
| `(visited_set, current_city)` | 加入一個未訪問城市 | Held-Karp DP | Exact DP，`O(n^2 * 2^n)` |
| 一整條完整 tour | swap / reverse 一段 | 2-opt / Local Search / SA | Heuristic，適合大 n |
| 城市排列 chromosome | crossover / mutation | GA | Population-based heuristic |
| OR-Tools routing variables | next variable / dimension constraints | Routing solver | 工程建模，支援 capacity / time window |

**會切換建模 = 真的懂 DSA**。

演算法不是單純套在題目名稱上，而是套在「你選擇的 state space」上。

## 範例 1：8-puzzle / 15-puzzle

### 基本建模

| 欄位 | 內容 |
|---|---|
| State | 棋盤排列，例如 3x3 array、string、tuple；需記錄空格位置 |
| Initial state | 題目給定的起始排列 |
| Action | 空格上下左右移動 |
| Transition cost | 每步 `1` |
| Objective | 最小化移動步數 |
| Constraints | 只能交換空格與相鄰格；不可越界；部分排列不可達 |
| Goal | 排列等於目標排列 |
| State-space size | `9!` 或 `16!` 級別，但只有一半排列可達 |
| Baseline | BFS 小規模、Dijkstra 等權退化成 BFS |
| Candidate algorithms | BFS、A*、IDA* |
| Guarantee | BFS 保證最短；A* 在 admissible heuristic 下保證最短 |

### 評估函數

常見 heuristic：

```text
h(state) = misplaced tiles
h(state) = Manhattan distance sum
```

`Manhattan distance` 通常比 `misplaced tiles` 強，因為它更細緻地估計每個 tile 距離目標位置還有多遠。只要不高估真實剩餘步數，就能保留 A* 的最佳性。

### 失敗風險

- 沒檢查 solvability，會對不可達狀態搜到爆。
- state encoding 不能 hash，visited set 失效。
- heuristic 過弱，A* 退化得接近 BFS。
- heuristic 若高估，A* 可能不再保證最短。

## 範例 2：TSP（Travelling Salesman）

### Prefix search 建模

| 欄位 | 內容 |
|---|---|
| State | 當前路徑 prefix，例如 `[0, 3, 2]` |
| Initial state | 起點城市 |
| Action | 選一個未訪問城市接到路徑尾端 |
| Transition cost | 新增邊的距離 |
| Objective | 最小化總 tour 長度 |
| Constraints | 每個城市剛好訪問一次，最後回到起點 |
| Goal | 訪問所有城市並回到起點 |
| State-space size | 約 `O(n!)` |
| Baseline | brute force 小 n、nearest neighbor |
| Candidate algorithms | Backtracking、Branch and Bound |
| Guarantee | 完整搜尋保證最佳；剪枝必須是 exact pruning 才保證 |

### DP 建模

| 欄位 | 內容 |
|---|---|
| State | `(visited_set, current_city)` |
| Action | 從 current_city 走到未訪問城市 |
| Objective | 最小化走到該 state 的累計距離 |
| Candidate algorithms | Held-Karp DP |
| Complexity | `O(n^2 * 2^n)` time，`O(n * 2^n)` space |
| Guarantee | Exact |

### Local Search 建模

| 欄位 | 內容 |
|---|---|
| State | 一條完整 tour |
| Action / Neighbor | swap、2-opt reverse、relocate |
| Objective | 最小化 tour length |
| Constraints | tour 必須包含所有城市且不重複 |
| Baseline | nearest neighbor initial tour、random tour |
| Candidate algorithms | Hill Climbing、2-opt、Simulated Annealing |
| Guarantee | 通常不保證最佳 |

TSP 是最適合練「同題多建模」的題目。小 n 用 exact 拿 optimum，大 n 用 greedy / 2-opt / SA，比較 gap 與 runtime。

## 範例 3：最短路（Dijkstra / A* 視角）

### Dijkstra 建模

| 欄位 | 內容 |
|---|---|
| State | 當前節點 |
| Initial state | source |
| Action | 走到相鄰節點 |
| Transition cost | 邊權重 |
| Objective | 最小化 source 到每個節點的距離 |
| Constraints | 邊權不可為負 |
| Goal | 單點最短路：到 target 可停；全源擴展：priority queue 空 |
| Baseline | BFS（所有邊權相同時） |
| Candidate algorithms | Dijkstra |
| Guarantee | 非負權重下保證最短 |

原筆記寫「state = 當前節點 + 已知最短距離」也可以，但要更精準地說：演算法狀態包含 priority queue 與 `dist[]`；搜尋節點本身是 vertex。

### A* 建模

| 欄位 | 內容 |
|---|---|
| State | 當前節點 |
| Action | 走到相鄰節點 |
| g(state) | source 到目前節點的已知成本 |
| h(state) | 目前節點到 target 的估計成本 |
| f(state) | `g(state) + h(state)` |
| Goal | pop 出 target 時停 |
| Guarantee | `h` admissible 且條件正確時保證最短 |

最短路題是判斷「需不需要 heuristic」的好例子。如果圖小或沒有自然座標，Dijkstra 可能已經足夠；若是 grid / map pathfinding，A* 才有明顯價值。

## 範例 4：N-Queens（Backtracking / Local Search 視角）

### Backtracking 建模

| 欄位 | 內容 |
|---|---|
| State | 已放好的皇后位置集合，通常是一列一列放 |
| Initial state | 空棋盤 |
| Action | 在下一列放一個不衝突的位置 |
| Cost | 純可行性問題，通常無轉移成本 |
| Objective | 找到任一可行解；或枚舉所有解 |
| Constraints | 任兩皇后不可同列、同欄、同斜線 |
| Goal | 放滿 `N` 個皇后 |
| Baseline | naive brute force |
| Candidate algorithms | Backtracking、Pruning、Bitmask optimization |
| Guarantee | 完整回溯保證找到解或證明無解 |

這個建模的關鍵是 action 只在「下一列」放一個皇后，直接消掉同列衝突，讓 constraints 檢查變成欄與兩條斜線。

### Local Search 建模

| 欄位 | 內容 |
|---|---|
| State | 每一列固定一個皇后，array `col[row]` |
| Action / Neighbor | 移動某一列的皇后到另一欄 |
| Objective | 最小化衝突 pair 數 |
| Constraints | 每列一個皇后；欄與斜線衝突可作為 penalty |
| Goal | 衝突數為 `0` |
| Candidate algorithms | Hill Climbing、Min-Conflicts、Random Restart |
| Guarantee | 通常不保證，但大 N 可很有效 |

同樣的 N-Queens，從「逐步構造」換成「完整解上修正」，就從 backtracking 變成 local search。

## 範例 5：Job Scheduling

Job Scheduling 是走向 heuristic / solver 前非常重要的工程題型，因為它有明確的 NP-hard 訊號：任務、機器、時間、依賴、優先級、deadline、capacity 一多，通常就不能期待簡單 exact 解。

### 單機最小化總完成時間

先看最簡版本：

| 欄位 | 內容 |
|---|---|
| State | 已排好的 job 序列 prefix |
| Initial state | 空序列 |
| Action | 選一個未排 job 接到序列尾端 |
| Transition cost | 新 job 完成時間對 objective 的貢獻 |
| Objective | 最小化總完成時間、最大 lateness、加權 tardiness 等 |
| Constraints | 每個 job 排一次；單機同時只能跑一個 job |
| Goal | 所有 job 都排完 |
| Baseline | 按 processing time、deadline、weight 排序 |
| Candidate algorithms | Greedy、DP 小規模、Branch and Bound |
| Guarantee | 視 objective 而定；某些簡化版本有 exact greedy |

這裡要小心：Job Scheduling 不是一題，而是一族問題。不同 objective 和 constraints 會讓問題難度完全改變。

### 多機 / 排班 / 工程實務建模

| 欄位 | 內容 |
|---|---|
| State | 每台機器目前的排程，或每個 job 的 assigned machine + start time |
| Action | 指派 job 到機器；調整 start time；交換兩個 job |
| Objective | 最小化 makespan、延遲懲罰、人力成本、換線成本 |
| Constraints | 機器 capacity、job dependency、deadline、skill、time window、休息規則 |
| Goal | 找到符合 constraints 的最低成本排程，或 time limit 內的可行好解 |
| Baseline | earliest deadline first、shortest processing time、list scheduling |
| Candidate algorithms | Greedy + local improvement、CP-SAT、MIP、SA |
| Guarantee | solver 視狀態可能回 `OPTIMAL` / `FEASIBLE` / `UNKNOWN`；手寫 heuristic 通常無最佳保證 |

工程上優先思路：

```text
小樣本 exact / brute force baseline
→ greedy baseline
→ CP-SAT / OR-Tools 建模
→ 必要時手寫 heuristic 或 hybrid
```

### 常見失敗

- 沒分清楚 hard constraint 和 soft penalty。
- objective 寫成單一分數，但權重完全沒校準。
- 只看 makespan，忽略 deadline / fairness / stability。
- solver 回 `FEASIBLE` 就當成最佳。
- heuristic 沒有和 greedy baseline 比，無法證明值得。

## 範例 6：工作問題建模：值班 / On-call 排班

這題貼近 Senior SE / DevOps / SRE 工作場景，也適合練 solver 思維。

### 問題描述

有一組工程師，要安排未來 4 週 on-call。每天需要 1 人值班。每個人有不可值班日期、負載上限、技能限制，且希望週末與夜間負擔公平。

### 建模

| 欄位 | 內容 |
|---|---|
| State | `assign[day] = engineer`，或 `x[day][engineer] ∈ {0,1}` |
| Initial state | 空排班，或沿用上週排班作為初始解 |
| Action | 指派某天給某工程師；交換兩天值班人；局部重排一週 |
| Objective | 最小化 unfairness、違反偏好的 penalty、連續值班 penalty |
| Hard constraints | 每天剛好 1 人；不可值班日期不能排；技能覆蓋必須滿足；每人上限不能超 |
| Soft constraints | 週末公平、不要連續值班、尊重偏好、少改既有班表 |
| Goal | 找到 hard constraints 全滿足、soft penalty 低的排班 |
| State-space size | 約 `engineers^days`，非常快爆 |
| Baseline | round-robin、greedy 按負載最低者指派 |
| Candidate algorithms | CP-SAT、local search、SA、greedy repair |
| Guarantee | CP-SAT 可在 time limit 內回可行 / 最佳狀態；手寫 heuristic 無最佳保證 |

### CP-SAT 變數視角

```text
x[d][e] = 1 表示第 d 天由工程師 e 值班

hard:
  for each d: sum_e x[d][e] = 1
  unavailable[d][e] => x[d][e] = 0
  for each e: min_load[e] <= sum_d x[d][e] <= max_load[e]

soft objective:
  minimize weekend_imbalance
         + consecutive_day_penalty
         + preference_violation_penalty
         + schedule_change_penalty
```

### Heuristic 視角

```text
initial:
  greedy round-robin，跳過 unavailable，優先負載低者

neighbor:
  swap two days
  reassign one day
  rotate one weekend block

cost:
  hard constraint violation * very_large_penalty
  + fairness penalty
  + preference penalty
  + stability penalty
```

這題最重要的是把 hard / soft 分開。Hard constraint 不能靠「分數看起來很高」混過去；soft constraint 才適合放進 weighted objective。

## 從建模到選算法

建模完成後，用下面這張表做第一輪選型。

| 建模特徵 | 優先考慮 |
|---|---|
| state 是 graph node，action 是走邊，cost 非負 | Dijkstra / A* |
| state 可由子問題組合，且有 optimal substructure | DP / memoization |
| state 是逐步構造解，constraints 可早期檢查 | Backtracking / pruning |
| state 是完整可行解，能定義 neighbor | Local Search / 2-opt / SA |
| objective 有可證明近似路線 | Approximation Algorithm |
| 問題像 routing / scheduling / assignment / packing | OR-Tools / CP-SAT / MIP |
| sequential decision，simulation 便宜，reward 可回傳 | MCTS / Monte Carlo |
| 解表示自然是排列、集合、bitstring，且需維持多樣性 | GA，但要有 baseline |

選型順序要符合 DSA README 的精神：

```text
先看能不能 exact
→ 再看是否有 approximation
→ 再看是否能用 solver
→ 最後才手寫 heuristic
```

## 評估函數怎麼設計

評估函數是 heuristic 的核心，但它也最容易被寫成玄學。

### 1. 先分 hard constraint 與 soft penalty

```text
bad:
  score = distance + unavailable_penalty + capacity_penalty

better:
  if violates hard constraints:
      reject state
  else:
      score = distance + soft_penalties
```

有些問題允許 infeasible state，例如 SA 早期探索或 repair heuristic。即使如此，也要明確標記哪些是 hard violation，不能讓它們和普通偏好混在一起。

### 2. 權重必須有意義

如果 objective 是：

```text
score = distance + 1000 * late_count + 3 * unfairness
```

就要能說出 `1000` 和 `3` 的理由。最少也要透過實驗比較不同權重下的結果分布。

### 3. 評估函數要能和 baseline 對話

heuristic 的報告至少要回答：

```text
random baseline:
greedy baseline:
exact optimum on small n:
heuristic best / average / worst:
gap to baseline:
runtime:
seed:
```

沒有這些，分數再漂亮也不知道是否有意義。

## Exact / Approximation / Heuristic 對同一模型的態度

| 類別 | 看待 state space 的方式 | 典型問題 |
|---|---|---|
| Exact | 完整搜尋或用數學結構壓縮搜尋 | 能不能保證最佳？複雜度多少？ |
| Approximation | 放棄最佳，但證明離最佳不會太遠 | 前提是否成立？近似比多少？ |
| Heuristic | 不保證最佳，用方向感與實驗換速度 | baseline 多少？分布穩不穩？ |
| Solver | 宣告變數、限制式、目標，交給求解器搜尋 | 模型是否正確？status / gap 怎麼解讀？ |

這也是為什麼「建模」比「背演算法名」重要。模型一變，保證就變；保證一變，工程責任也變。

## 常見建模錯誤

| 錯誤 | 後果 | 修正 |
|---|---|---|
| State 少記資訊 | 不同狀態被誤認成同一個，答案錯 | 檢查 state 是否足以決定後續可行 action |
| State 記太多資訊 | visited / DP key 爆炸 | 移除可由其他欄位推導的資訊 |
| Action 太粗 | 找不到細緻改善路徑 | 拆成更小 transition 或 neighbor |
| Action 太細 | 分支度過大，搜尋爆炸 | 加 pruning、限制候選集合 |
| Objective 混入 hard constraint | 看似低分但不可用 | hard constraint 先檢查，soft penalty 再加權 |
| 沒有 baseline | 不知道 heuristic 是否有效 | 建 random / greedy / exact 小樣本 |
| 沒說 guarantee | 對外承諾錯誤 | 明確標記 optimal / approximate / heuristic / feasible |
| 終止條件模糊 | 可能無限跑或過早停 | 設 iteration、time limit、no-improvement limit |

## 練習方式

每攻完一個 ⭐ 或 ✦ topic 後，回來這份筆記補一題：用該 topic 的視角重新建模一個問題。

最低要求：

- 用 `state + action + objective + constraints` 寫完。
- 說清楚是否保證最佳解。
- 至少給一個 baseline。
- 如果是 heuristic，記錄 seed、best / average / worst、runtime。
- 如果是 solver，記錄 status、time limit、objective value、gap。

## 結論

建模是 heuristic 前的門檻，也是 solver 前的門檻。

一個成熟的最佳化問題處理流程，不是先問「要不要用什麼聰明算法」，而是先問：

```text
state 是否完整
action 是否合理
objective 是否反映真正價值
constraints 是否分清 hard / soft
baseline 是否存在
guarantee 是否講清楚
```

這六件事過關，後面選 exact、approximation、heuristic 或 solver 才有意義。
