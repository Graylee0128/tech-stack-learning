# 從啟發式搜尋到近似最佳化（橫縱分析報告）

> 研究時間：2026-05-06 | 所屬領域：DSA / Combinatorial Optimization | 研究對象類型：技術概念群
> 報告 2 主文。範圍：heuristic search、local search、metaheuristics、Monte Carlo、approximation、solver。
> 不重講報告 1 的 DSA 基本功，只處理從「能 exact 解」走到「必須接受近似、啟發式與 solver」之後的世界。

## 一句話定義

啟發式搜尋到近似最佳化這條線，研究的是：當問題大到不能硬求最佳解時，工程師如何在「速度、解品質、可證明性、可維運性」之間選一個能交付的解法。

它不是一組演算法清單。

它更像一套判斷語言。A*、2-opt、Simulated Annealing、MCTS、Set Cover greedy、OR-Tools 都在這張圖裡，但它們不是同一類東西。有些是在搜尋時加方向感，有些是在可行解附近修修補補，有些是拿隨機性換探索，有些是用數學證明換誤差上界，有些則把多年 OR 與 SAT/CP/MIP 技術包成工業級工具。

這也是 AI 時代 Senior SE 需要補這條線的原因。不是因為每個人都該手寫 GA 或 SA，而是因為 AI 會很容易產生「看起來聰明」的最佳化程式。你要能看出它是在 exact、approximation、heuristic 還是 solver 的哪個世界裡說話；它保證了什麼、放棄了什麼、實驗該怎麼驗。

## 縱軸：從精確計算到可交付近似

### 1950s-60s：OR 的底色，最佳化先是數學建模問題

這條線的源頭不是 LeetCode，也不是 AI。

它更早出現在 Operations Research 裡。戰後工業、軍事、物流與資源分配問題，逼著人們把「怎麼排、怎麼分、怎麼走、怎麼省」寫成數學模型。Linear Programming、Simplex、Dynamic Programming 在這個背景下成熟起來。這一代人的核心信念很清楚：先把問題寫成模型，再談怎麼求解。

這個底色到今天都沒有退場。

你在 OR-Tools 或 Gurobi 裡寫 constraints，本質上仍然是在繼承這個傳統。差別只是以前可能手算或跑專門程式，現在你用 Python API 宣告變數、限制式、目標函數，然後把搜尋、剪枝、鬆弛、衝突學習交給 solver。

對 DSA 學習來說，這一段留下的遺產是：最佳化不是「想到一個聰明技巧」，而是先把問題壓成一個清楚的形式。

### 1968：A* 把「直覺」變成可分析的搜尋導引

1968 年，Hart、Nilsson、Raphael 發表 A* 的經典論文。這件事的重要性不只是多了一個 pathfinding algorithm。

在 A* 之前，人們當然也會用各種 ad hoc 的方式讓搜尋更快。比如「看起來離目標比較近的點先走」。但 A* 給了這種直覺一個乾淨的語言：

```text
f(n) = g(n) + h(n)
```

`g(n)` 是已經付出的成本，`h(n)` 是到目標的估計成本。只要 `h` 滿足 admissible 這類條件，A* 就不只是「通常比較快」，它還保留最短路保證。

這是 heuristic search 最漂亮的地方：heuristic 不一定等於放棄正確性。A* 這條支線告訴我們，啟發函數如果設計得夠克制，它可以只是幫 exact search 排順序，而不是剪掉答案。

但它也埋下第一個工程問題。`h` 太弱，A* 退化得像 Dijkstra；`h` 太激進，速度可能上去，最佳性可能下來。後來 Weighted A*、Beam Search 的取捨，都在這個張力上展開。

### 1970s：NP-hard 讓人承認「最佳」不是永遠買得起

1970s 是整條線的心理轉折。

Karp 在 1972 年整理出一批經典 NP-complete 問題，把很多看似不同的難題放到同一張地圖上。這不是單純的理論分類，而是在告訴工程師一件不太舒服的事：有些問題不是你不夠努力，也不是程式寫得不夠漂亮，而是目前沒有已知方法能在一般情況下又快又保證最佳。

這個認知一出現，演算法世界的語氣就變了。

以前問的是「最優解在哪裡？」後來多了一串更成熟的問題：

- 我能不能只在小樣本上 exact，拿來當 baseline？
- 我能不能接受一個可證明的近似比？
- 我能不能接受沒有保證，但實務上穩定夠好？
- 我能不能把問題丟給 solver，設 time limit，拿 feasible 或 near-optimal solution？

這些問題構成報告 2 的主線。

#### NP-hard 之後：現代研究多半是在側面攻破

一旦承認一般情況下「又快又保證最佳」買不起，研究方向就很少再正面宣稱自己把 NP-hard 問題完全破解。更常見的是從側面攻破：放寬某個條件，換回可交付的保證或實務效果。

- 放寬最佳性：approximation 不一定回傳最佳解，但給你近似比。
- 放寬確定性：randomized algorithms 用機率換速度、簡化或探索能力。
- 放寬一般性：parameterized algorithms 問「如果某個參數很小，能不能變得可解」。
- 放寬最壞情況：solver 不承諾所有 instance 都輕鬆，但利用剪枝、relaxation、constraint propagation 與 heuristic，在真實資料上常常很強。
- 放寬完整搜尋：heuristic / metaheuristic 不保證最壞情況，但用 baseline、實驗設計與監控換取工程可用性。

所以 NP-hard 不是終點，而是分流點。它把問題從「找一個漂亮公式」推向更成熟的提問：我要犧牲哪一種保證，換哪一種可用性？

### 1970s-90s：近似算法建立「放棄最佳，但不放棄證明」的世界

Approximation Algorithms 是很多人最容易跟 heuristic 混在一起的地方。

它們都可能不給最佳解，但心態完全不同。Heuristic 常說「我實驗上跑得不錯」。Approximation 則說「我可以證明這個解離最佳解最多多遠」。

Set Cover 的 greedy approximation 是很好的入口。每次選單位成本覆蓋最多新元素的集合，看起來像直覺貪心；Chvatal 1979 的結果則告訴你，這個直覺背後有 logarithmic ratio。Christofides 對 metric TSP 的方法也很經典：同樣叫 heuristic，但它有小於 `3/2` 的 worst-case ratio。這類結果把「工程上看起來合理」推進到「數學上可以交代」。

近似算法的價值不在於它永遠最好用。很多 approximation 在實務上常數不小、實作不輕、模型限制也強。它真正提供的是談判語言：當你面對 NP-hard 問題，又不能只用「我感覺還行」交差時，approximation 是一條介於 exact 與 heuristic 之間的中間路。

1990s 以後，近似算法、隨機算法、參數化算法與 SAT/CP/MIP solver 的研究，把焦點從「一定要最佳」轉向「可證明地接近最佳、可證明地利用結構，或在實務上有效」。Approximation 問「離最佳最多多遠」；randomized algorithms 問「用機率能不能換取速度或簡化」；parameterized algorithms 問「如果某個參數很小，能不能把難題變得可解」；solver 則問「能不能用模型與工業級搜尋策略，在真實 instance 上拿到可用答案」。這四條線共同把 NP-hard 之後的世界拆得更細，不再只剩 exact 與 heuristic 二分法。

### 1980s-90s：Local Search 與 Metaheuristic 把「可行解」當成地形

Local Search 的世界觀很不一樣。

它不是從起點開始一層一層擴展搜尋樹，而是先拿到一個完整可行解，再問：附近有沒有更好的解？

TSP 的 2-opt 是最乾淨的例子。你先有一條 tour，然後反覆檢查能不能用兩條邊交換讓路徑變短。這不保證全域最佳，但直覺強、實作短、效果常常驚人。它代表一種工程上很常見的策略：先用 greedy 做出初始解，再用 neighborhood move 慢慢修。

問題也很清楚。Local Search 容易卡在 local optimum。於是 1980s 之後，Simulated Annealing、Tabu Search、Genetic Algorithm、Ant Colony Optimization 等 metaheuristic 開始流行。它們共同面對的是同一件事：怎麼不要太早被眼前的好解困住。

SA 用溫度讓壞 move 在早期也有機會被接受，後期逐步收斂。Tabu Search 用記憶避免在同一批狀態來回打轉。GA 用 population、selection、crossover、mutation 維持多樣性。ACO 用 pheromone 類比讓好路徑被強化。

這些方法很有魅力，也很危險。

魅力在於它們不要求問題長得很標準，只要你能定義 solution、neighbor 或 fitness，通常就能開工。危險在於它們很容易變成參數工地：temperature、cooling rate、mutation rate、population size、pheromone evaporation，每個都看似可調，每個都可能讓結果失真。

所以學 metaheuristic 的重點不是背名字，而是學會三件事：

- 解的表示法是否自然
- 鄰域或變異是否真的能探索到重要解
- 實驗是否能用 seed、baseline、分布統計重現

沒有這三件事，再華麗的 heuristic 都只是在賭。

### 2000s-2010s：Monte Carlo 與 MCTS 讓抽樣成為搜尋策略

Monte Carlo 方法引入的是另一種妥協：不完全展開，也不完全相信手寫 evaluation，而是用抽樣估計價值。

MCTS 的成名和圍棋關係很深。Coulom 2006 把 Monte Carlo evaluation 與 tree search 結合得更系統；Kocsis 與 Szepesvári 同年提出 UCT，把 bandit ideas 放進 Monte Carlo planning。這條線的力量在於，它可以在巨大狀態空間裡把計算量集中到有希望的分支上。

AlphaGo 讓這件事進入主流視野。2016 年的 Nature 論文不是單靠 deep learning，也不是單靠 tree search，而是 policy/value networks 與 Monte Carlo tree search 的混合。這個案例很適合放進 DSA 學習地圖，因為它提醒我們：AI 系統裡的「聰明」常常不是單一模型，而是模型、搜尋、抽樣、工程限制合在一起。

MCTS 也不是萬靈丹。它適合 sequential decision、game tree、simulation 相對便宜、reward 能回傳的問題。如果 rollout 很貴、reward 極度稀疏、狀態轉移難以模擬，它就會變得笨重。

這裡的判斷點很實際：你不是看到「決策問題」就用 MCTS，而是先問自己能不能便宜地模擬很多次。

### 2010s-2020s：Solver 工程化，手寫 heuristic 不再是預設答案

到了 OR-Tools、CP-SAT、Gurobi、HiGHS、MiniZinc、Z3 這個時代，一件事變得很明顯：很多工程問題不該從手寫 heuristic 開始。

Google OR-Tools 的官方文件把自己定位成 combinatorial optimization suite，覆蓋 routing、flows、integer/linear programming、constraint programming。Routing 文件也講得很坦白：VRP 會隨規模快速變難，對大問題 solver 有時會回傳 good but not optimal 的解；但它支援 TSP、VRP、capacity、time windows、dropped visits 等常見變體。這正是工程上最需要的東西。

Solver 的世界觀是：你先把問題建模，然後讓工具在內部混合 exact search、relaxation、cut、constraint propagation、heuristics。它可能給你 `OPTIMAL`，也可能只給 `FEASIBLE`，甚至因 time limit 回 `UNKNOWN`。成熟工程師要讀得懂這些狀態，而不是把 solver 當黑盒神諭。

這也改變學習順序。

今天學 heuristic，不是為了每題都手寫一個。更好的順序是：

```text
小樣本 exact baseline
→ greedy baseline
→ 判斷是否有 approximation
→ 判斷是否能丟 solver
→ 必要時才手寫 heuristic
→ 用實驗分布驗證
```

這條路徑比較慢，但比較像真實工作。

## 橫軸：當下時間切面的對比

把同一組技術放到不同世界裡，會看到完全不同的價值排序。

| 世界 | 重視什麼 | 不重視什麼 | 對 heuristic 學習的意義 |
|---|---|---|---|
| OR / Operations Research | 建模、理論保證、可證明界線、solver 能力 | 炫技式手寫 | Approximation、LP/MIP、CP-SAT 是核心語言 |
| AI / Game Playing | 抽樣效率、search-policy-value 混合、長期 reward | 傳統最佳性證明 | MCTS / RL 的重要性高，但要看 simulation 成本 |
| 工程 / 排班 / 物流 | 可維運、能加 constraint、能設 time limit、能解釋結果 | 學術新穎度 | 優先 solver；手寫 heuristic 是補位，不是信仰 |
| 競賽 / 演算法社群 | 嚴謹證明、最壞情況、資料範圍、可 AC | 黑盒 solver、隨機調參 | 提供 exact 與 approximation 地基，但不代表工程全貌 |

這張表最想講的是：同一個技術，在不同場域裡的地位會變。

A* 在遊戲與路徑規劃裡是主角，在一般排班裡可能只是小工具。MCTS 在圍棋裡很耀眼，在普通物流 routing 裡通常不是首選。Approximation 在理論上優雅，但工程上不一定比 solver 好落地。Solver 在競賽裡像作弊，在排班後台裡卻是正道。

學習時不要問「哪個演算法最強」。要問「這個世界用什麼標準判斷強」。

## 交叉分析：六大派別的本質區分

這份報告的核心，是把常被混在一起講的東西分清楚。

| 派別 | 核心方法 | 最佳性保證 | 理論誤差上界 | 典型代表 | 最該學的判斷點 |
|---|---|---|---|---|---|
| Heuristic Search | 用啟發函數導引搜尋順序 | A* 在 admissible `h` 下可保證 | 通常沒有一般近似比 | A* / IDA* / Weighted A* / Beam Search | `h` 是保守估計、激進估計，還是純排序分數 |
| Local Search | 從完整可行解出發，在鄰域內改良 | 否 | 否 | Hill Climbing / 2-opt / k-opt | state 表示與 neighbor 設計是否抓到問題結構 |
| Metaheuristic | 用隨機性、記憶或族群跳脫 local optimum | 否 | 否 | SA / Tabu / GA / ACO | 參數、seed、結果分布是否可重現 |
| Monte Carlo | 用抽樣估計 value 或 policy | 否 | 有機率收斂或採樣誤差語言 | Monte Carlo Simulation / MCTS / UCT | simulation 是否便宜，reward 是否可靠 |
| Approximation | 多項式時間演算法加上近似比證明 | 否 | 有，例：`2-approx`、`O(log n)`、`3/2` | Set Cover greedy / Vertex Cover 2-approx / Christofides | 問題是否符合證明前提 |
| Solver | 用建模介面包裝多種求解策略 | 依狀態而定 | 依模型與 solver 而定 | OR-Tools / Gurobi / CP-SAT / Z3 | `OPTIMAL`、`FEASIBLE`、`UNKNOWN` 代表什麼 |

最容易出錯的是把「有 heuristic 味道」誤認成「沒有保證」。

A* 的 `h` 是 heuristic，但在條件滿足時能保證最短路。Christofides 常被叫 heuristic，但它對 metric TSP 有近似比。OR-Tools 裡也會用 heuristic，但 solver 回傳狀態時仍有嚴格語意。反過來，GA 聽起來很有自然智慧，如果沒有 baseline 與統計，它可能只是很貴的 random search。

派別不是身份標籤，而是風險標籤。

你選了哪一派，就選了哪種風險：

- Heuristic Search 的風險是 `h` 設計錯
- Local Search 的風險是卡在局部最優
- Metaheuristic 的風險是參數與實驗不可驗證
- Monte Carlo 的風險是抽樣成本與估計方差
- Approximation 的風險是前提太窄或實務常數太大
- Solver 的風險是模型寫錯、狀態讀錯、time limit 下誤以為 optimal

這些風險比演算法名字更重要。

## 橫縱交匯洞察

### 洞察 1：歷史不是淘汰賽，而是分層堆疊

很多技術學習地圖會把演算法排成「從舊到新」的進步故事，好像 solver 出現後手寫 heuristic 就過時，或 AlphaGo 出現後傳統搜尋就退場。

真實情況比較像地層。

OR 的建模語言還在 solver 裡。A* 的 `g + h` 思想還在遊戲 AI 與路徑規劃裡。NP-hard 的理論邊界還在提醒你別亂承諾最佳解。Local Search 的 neighborhood 還在 routing solver 裡當內部策略。Monte Carlo tree search 在 deep RL 裡被重新接上神經網路。

每一層都沒有完全消失，只是退到不同位置。

所以這份 README 的取捨很正確：不要一開始就跳 GA、ACO、MCTS。你還沒有 state/action/objective/constraints 的語言時，學那些只會變成背名詞。先把 exact、greedy、DP、graph、search 的地基補起來，後面的東西才不是玄學。

### 洞察 2：AI 時代更需要 baseline，不是更少

LLM 很會產生看似合理的 heuristic。它會自然地寫出 scoring function、magic weights、local adjustment、random restart，甚至給你一段漂亮的註解。

這很方便，也很危險。

在 heuristic 世界裡，程式能跑不代表它有意義。你至少要問：

- random baseline 是多少？
- greedy baseline 是多少？
- 小樣本 exact optimum 是多少？
- 同一 seed 能不能重現？
- 跑 100 次的 best / average / worst / variance 長什麼樣？
- 規模變大時 runtime 與解品質怎麼變？

沒有 baseline，heuristic 的敘事會太自由。AI 生成的程式尤其如此，因為它常能把一個沒有驗證的想法寫得很像工程經驗。

報告 2 的學習主軸因此不只是「學 heuristic」，而是「學會驗 heuristic」。

### 洞察 3：Solver 是工程預設，手寫 heuristic 是有理由的例外

在真實工程裡，排班、派工、routing、assignment、packing 這些問題常常不是缺一個聰明演算法，而是缺一個穩定模型。

OR-Tools 這類工具的優勢不只是解品質。它更大的價值是把需求變更變成 constraint 變更。今天加 time window，明天加 capacity，後天允許 dropped visits 並給 penalty；如果你手寫 heuristic，這些需求很容易把原本漂亮的程式拖成一團條件分支。用 solver，至少你有機會把變更留在模型層。

但手寫 heuristic 還是有位置。

當問題太特殊、延遲要求太低、規模太大、部署環境不能吃重依賴，或 solver 模型反而難以表達時，手寫才變成合理選擇。這時最常見的成熟路線也不是從零發明一個 metaheuristic，而是：

```text
greedy initial solution
→ local improvement
→ random restart / SA escape
→ solver 或 exact 小樣本校準
```

手寫 heuristic 的專業感，不在於名字華麗，而在於它跟問題結構貼得很緊。

### 洞察 4：Approximation 是「理論保證」與「工程可用」之間的橋，不是 heuristic 的同義詞

Approximation 最常被低估。

很多工程師會把「不求最佳」全部丟進 heuristic 盒子裡。這會失去一種很重要的選項：你可以放棄最佳，但保留誤差界線。

Set Cover、Vertex Cover、Metric TSP 這些例子都在提醒我們，某些 NP-hard 問題不是只能在 exact 與亂試之間二選一。你可以用 polynomial-time 的方法拿一個可證明接近最佳的解。

當然，approximation 不是萬能。它常受限於問題前提，實作也未必最短。可是一旦需求裡出現「不能保證最佳沒關係，但要能對上層交代最壞差多少」，approximation 就比純 heuristic 更適合當主角。

### 洞察 5：未來的能力不是「手寫所有算法」，而是「讀懂求解系統」

未來最值錢的能力不會是把每個 metaheuristic 都從零默寫一次。那比較像修煉，不像工作。

更實用的能力是讀懂求解系統：

- 讀懂問題是不是 NP-hard 訊號
- 讀懂目標函數與 constraint 是否寫對
- 讀懂 solver 狀態與 gap
- 讀懂 heuristic 的實驗設計
- 讀懂 approximation 的前提
- 讀懂 AI 生成解法在哪裡偷換了保證

這也是報告 2 跟報告 1 的接點。報告 1 讓你能讀懂演算法。報告 2 讓你能讀懂取捨。

## Topic 取捨原則

對應 `../../README.md` 的學習順序，這條線不應該展開成無限大全家桶。先收能建立判斷力的主線。

### 主線必學

| Topic | 為什麼收 | 最小完成標準 |
|---|---|---|
| Baseline 思維 | heuristic 沒 baseline 就不可驗證 | 同一題能做 random / greedy / exact 小樣本 |
| A* / Weighted A* / Beam Search | 最乾淨的 heuristic search 對照組 | 用 Grid Pathfinding 比 BFS / Dijkstra / A* |
| Hill Climbing / Random Restart / 2-opt | Local Search 入門，TSP 可直接實驗 | 會設計 neighbor，會報 best/avg/worst |
| Simulated Annealing | 第一個 metaheuristic，能學 exploration vs exploitation | 會解釋 temperature schedule 與失敗模式 |
| Monte Carlo Simulation / Randomized Algorithms | 為 MCTS 與抽樣思維打底 | 會用 seed 重現，會看 variance |
| NP-hard Recognition | 選型前置雷達 | 看到 routing / scheduling / subset / assignment 能警覺 |
| Approximation Algorithms | 補上可證明近似的中間路 | 至少懂 Set Cover greedy、Vertex Cover 2-approx、Metric TSP |
| OR-Tools Routing / CP-SAT | 工程最常用入口 | 會建模小型 VRP / scheduling，讀懂 solver status |

### 延伸支線

| Topic | 何時再學 |
|---|---|
| Genetic Algorithm | 需要 population-based search，且 solution encoding 很自然 |
| Ant Colony Optimization | routing 類研究或需要了解歷史脈絡時 |
| Tabu Search | Local Search 已熟，想加 memory mechanism 時 |
| MCTS | 問題是 sequential decision，且 simulation 便宜時 |
| Gurobi / CPLEX / HiGHS | 工作開始碰 LP/MIP，且 OR-Tools 不夠時 |
| MiniZinc / Z3 | constraint 或 logic-heavy 問題變多時 |

### 暫不收主線

Segment Tree、HLD、SCC、Network Flow 等高階 DSA 競賽支線不是 heuristic 進階。它們很重要，但應由具體題目需求觸發。尤其 Network Flow 容易被誤放進這條線：它是很多最佳化問題的 exact polynomial-time 解法，不是 heuristic 的後續章節。

## 實驗設計：每個 heuristic topic 都要留下什麼

報告 2 的每個 topic 筆記，不只寫概念，也要能跑實驗。最低紀錄格式如下：

```text
problem:
  input scale:
  random seed:
  objective:
  constraints:

baselines:
  exact small n:
  greedy:
  random:

algorithm:
  state representation:
  action / neighbor:
  evaluation:
  parameters:

results:
  best:
  average:
  worst:
  standard deviation:
  runtime:
  optimality guarantee:
  gap to baseline:

failure notes:
  where it gets stuck:
  which parameter is sensitive:
  what breaks when n grows:
```

這份格式看起來有點重，但它會救你。

因為 heuristic 最怕的不是慢，而是你不知道它為什麼快、為什麼好、哪天會壞。

## 決策樹：拿到最佳化問題時怎麼選

```text
1. 能否用已知 exact algorithm 在合理時間內解？
   是 → 用 exact。例：Dijkstra、min cost flow、matching、DP 小規模。
   否 → 繼續。

2. 問題是否有 NP-hard 訊號？
   否 → 先回報告 1 找資料結構 / 圖 / DP / greedy。
   是 → 繼續。

3. 是否是標準 routing / scheduling / assignment / packing / CP / MIP？
   是 → 優先 OR-Tools / CP-SAT / MIP solver，設 time limit 與 baseline。
   否 → 繼續。

4. 是否有可接受的 approximation algorithm？
   是 → 若需要誤差交代，優先 approximation。
   否 → 繼續。

5. 是否能自然定義完整解與 neighbor？
   是 → Local Search / 2-opt / SA。
   否 → 繼續。

6. 是否是 sequential decision 且 simulation 便宜？
   是 → Monte Carlo / MCTS。
   否 → 客製 greedy + problem-specific heuristic，並強制做 baseline。
```

這棵樹的精神很簡單：先找保證，再找工具，再找手寫。

## 建議學習路線

### 第一輪：把判斷工具補齊

- NP-hard 識別
- Baseline 思維
- Exact / Approximation / Heuristic 差異
- `state + action + objective + constraints`

這一輪不追求演算法多。目標是看到題目時能判斷「這題不該硬求最佳」。

### 第二輪：用同一題做橫向比較

選 Grid Pathfinding 與 TSP 兩個題型：

- Grid：BFS / Dijkstra / A* / Weighted A*
- TSP：brute force 小樣本 / nearest neighbor / 2-opt / SA

同一題多解，比單獨學四個演算法有效很多。你會看見它們犧牲的是什麼。

### 第三輪：補 approximation 與 solver

- Set Cover：brute force 小樣本 / greedy approximation
- Vertex Cover：exact 小樣本 / 2-approx
- VRP 或排班：handwritten heuristic / OR-Tools

這一輪的目標是建立工程眼光：不是每個問題都值得自己寫搜尋。

### 第四輪：只在需要時加支線

MCTS、GA、ACO、Tabu Search 不急著排進主線。它們每個都能學很深，但在目前 DSA 學習計畫裡，應該由具體題目觸發。

## 結論

從啟發式搜尋到近似最佳化，真正要學的不是一串演算法名字，而是面對不可硬解問題時的成熟姿勢。

你要知道什麼時候最佳解仍然便宜，什麼時候只能在小樣本上 exact；什麼時候該要求近似比，什麼時候該接受實驗性 heuristic；什麼時候該手寫，什麼時候該把尊嚴收起來交給 solver。

報告 2 的主線因此可以收束成一句話：

> 先建模，再判斷保證；先 baseline，再談 heuristic；先找 solver，再決定要不要手寫。

這句話比任何單一演算法都重要。
