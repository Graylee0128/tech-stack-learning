# SRE 視角下的效能、容量與監控教學筆記

> 這份筆記以 `study-area.sre.tw` 的 `R1: Site Reliability Engineering` 為主，重點不是做整本書摘要，而是把跟 Go 後端工作最有關的部分抽出來：`SLI/SLO`、monitoring、capacity planning、toil / automation、on-call。讀完之後，你會比較知道「為什麼要學 R4」，而不只是知道 Linux 底層長什麼樣子。

## 先講結論

R1 在教你的不是某一套工具，而是一種判斷方式：

> 可靠性、效能、容量、監控、事故處理，從來不是五件事，而是同一件事的五個面向。

如果你未來做的是博彩或電商後端，這個觀念非常重要。  
因為線上系統真正可怕的，不是功能寫不出來，而是流量一來時整個系統開始失真：

- 使用者說變慢了，但 dashboard 看不出來
- error rate 不高，但實際體驗已經壞了
- 加機器沒有用，因為壓力都打在共享資源
- 每次大促或大賽都靠人肉盯盤和手動救火

R1 的價值，就是幫你從「會寫服務」變成「知道怎麼讓服務長期穩定」。

## 1. SRE 的上位視角是什麼

### 觀念

`CH01` 把 SRE 的主線列得很清楚：

- 長期把工程能力放在可靠性問題上
- 在不違反 SLO 的前提下最大化變更速度
- 監控系統
- 緊急事件處理
- 變更管理
- 需求預測與容量規劃
- 資源部署
- 效率與性能

這段很關鍵，因為它直接說明：

SRE 不是「值班的人」，也不是「幫忙修機器的人」。  
SRE 是用工程方法處理可靠性的人。

### 白話比喻

如果把產品團隊想成在經營一家很大的商場：

- backend engineer 像是在蓋店、修店、加功能
- SRE 更像是在確保商場在尖峰時段不會全館斷電、不會塞爆、不會每次活動都靠人衝現場救火

### 工程現象

一個團隊如果沒有 SRE 視角，常常會出現：

- 新功能一直上，但監控跟不上
- 效能問題都等線上爆了才知道
- capacity 沒規劃，只能事後加機器
- on-call 很痛苦，因為警報沒有經過設計

### Go 實務映射

你做 Go 服務時，真正成熟的目標不是「handler 有回 200」，而是：

- response time 能不能長期穩定
- 尖峰流量時 error rate 會不會飆
- 部署時會不會傷到 SLO
- incident 發生時有沒有足夠資訊判斷

## 2. 為什麼 SLI / SLO 會決定你怎麼看效能

### 觀念

R1 的 `CH01` 提到 SLO 是主線；R4 的 `01 如何學習 Linux 效能最佳化` 甚至直接把效能分析的第一步和第二步寫成：

1. 選擇指標，也就是 `SLI`
2. 設定目標，也就是 `SLO`

這件事非常重要。因為如果你沒有先定義「什麼叫好」，那你看到再多指標也不知道是不是問題。

### 白話比喻

如果老闆只說「網站要快一點」，這句話幾乎沒有操作性。  
但如果換成：

- 99% 下單請求在 300ms 內完成
- 錯誤率低於 0.1%

團隊就知道什麼叫正常、什麼叫退化。

### 常見誤解

- `SLO 是給 SRE 看的，backend 不用管`  
  錯。你寫的 timeout、cache、SQL、重試，都會直接影響 SLO。
- `CPU 低就是健康`  
  錯。真正重要的是使用者有沒有感受到服務可用。

### Go 實務映射

如果你做的是博彩或電商，常見 SLI 可能包含：

- API latency
- 下單成功率
- 查詢盤口或商品頁的錯誤率
- WebSocket 訊息延遲
- DB / Redis 依賴可用性

你要先知道自己的服務對產品承諾的是哪個指標，後面才知道要優化哪裡。

## 3. Monitoring 不是有 dashboard 就好

### 觀念

`CH05-07` 裡對 monitoring 的討論其實很實戰，問題都很到位：

- white-box 與 black-box 差在哪
- 除了四大黃金指標，還要看什麼
- alert、dashboard、指標收集是否應相對獨立
- 要不要安排人盯 dashboard

這些問題的共同核心是：

> 監控不是把圖畫出來，而是讓團隊在系統偏離正常狀態時能及早發現、定位和行動。

### white-box 和 black-box

- `white-box`: 看系統內部，例如 CPU、goroutine、queue length、DB pool usage
- `black-box`: 從外部看服務是否正常，例如實際 HTTP 探測、登入流程、下單流程

兩者都要有。

### 白話比喻

只看 white-box，像是你只看餐廳後廚儀表板，卻不知道客人有沒有真的吃到菜。  
只看 black-box，像是你只知道客人在抱怨，但不知道是爐火壞了、外場塞住，還是食材沒到。

### 常見誤解

- `有 Grafana 就叫有監控`  
  錯。沒有可行動的告警與對應指標，圖再多也只是裝飾。
- `把所有數據都收起來就好了`  
  錯。沒有分層與取捨，最後只會告警疲勞。

### Go 實務映射

Go 服務可以很自然地補這些內部訊號：

- request latency histogram
- error count
- goroutine 數量
- GC pause
- DB connection pool 使用率
- queue backlog

但還要補上 black-box，例如：

- 從外部定時打商品頁、下單 API、登入 API
- 對 WebSocket 做端到端延遲探測

## 4. Capacity planning 為什麼和高併發是同一題

### 觀念

R1 `CH01` 把 `Demand Forecasting and Capacity Planning` 放進 SRE 主線裡，這不是附屬題，而是核心題。  
補充材料 `SRE_CH18` 也在談容量規劃如何用更抽象但更能服務業務的方式來思考資源需求。

### 白話比喻

容量規劃不是在問「現在 CPU 還剩幾顆」，而是在問：

- 活動或比賽開始後，流量會怎麼長
- 哪一段路最先塞住
- 需要保留多少餘裕
- 某一台或某一區故障時，剩下的還撐不撐得住

### 工程現象

在博彩或電商，容量問題通常不只是 Web instance 數量，而是整條鏈：

- app pods
- DB max connections
- Redis 吞吐
- MQ 消費速度
- LB / firewall / NAT 連線表
- WebSocket 長連線數

### 常見誤解

- `容量規劃 = 每次快不夠時加機器`  
  那叫事後補洞，不叫規劃。
- `有 autoscaling 就等於容量規劃完成`  
  錯。很多共享依賴不能跟著線性擴。

### Go 實務映射

當你做 Go 服務時，容量規劃常常會變成這些問題：

- 單個 instance 能撐多少 RPS / 連線數？
- DB pool 設到多少才合理？
- Redis 熱 key 會不會把單點打爆？
- 某個熱門 endpoint 是否需要 cache 或預熱？

這就是為什麼 R4 的高併發知識和 R1 的容量規劃其實是同一題的兩面。

## 5. Toil 與 automation 的重點，不是自動化很酷

### 觀念

`CH05-07` 在談 `toil`、monitoring、Google 自動化系統演進。裡面的問題非常貼近現場：

- 手動跑腳本算不算 toil
- 什麼時候應該自動化
- 自動化做到什麼程度
- 維護自動化本身要花多少成本

真正要記住的是：

> 自動化的目的不是把人移除，而是把重複、容易出錯、沒有工程價值的操作從人手上拿掉。

### 白話比喻

如果每次活動前都要有人半夜手動改設定、重啟服務、清 cache、盯 log，這種團隊遲早會累垮。  
好的自動化不是炫技，而是讓系統不再依賴某個最熟的人在關鍵時刻在線。

### 常見誤解

- `只要手動做過一次，就值得自動化`  
  不一定。要看頻率、風險、維護成本。
- `自動化腳本只要能跑就好`  
  錯。腳本如果沒有版本控制、沒有 review、沒有觀測，最後會變成新的 toil。

### Go 實務映射

Go 團隊很常寫：

- maintenance job
- consumer / worker
- migration tool
- internal admin API

這些東西如果只追求「能做事」，最後會把運維風險埋進系統裡。  
SRE 視角會逼你多問兩題：

- 這個流程是不是該標準化？
- 這個工具出了問題，我們能不能觀測、回滾、限權？

## 6. Release、Simplicity、Alerting、On-Call 其實是一組

### 觀念

`CH08-11` 表面上看起來是四個主題：

- Release Engineering
- Simplicity
- Practical Alerting from Time-Series Data
- Being On-Call

但它們其實在說同一件事：

> 如果系統複雜、部署風險高、告警無法行動、值班制度爛，那線上穩定性遲早會崩。

### Release Engineering

重點不是「能部署」，而是：

- 發佈頻率
- 發佈風險
- 發佈過程是否可觀測
- 權限控管與回滾是否清楚

對高流量服務來說，部署本身就是風險事件。

### Simplicity

複雜度本身就是可靠性成本。  
每多一個特例、多一層 workaround、多一個只有某個人看得懂的腳本，都會在事故時放大。

### Alerting

好的 alert 不是「一有異常就叫」。  
好的 alert 是：

- 真正偏離正常狀態時會叫
- 叫了之後知道誰該做什麼
- 不會把團隊吵到對告警麻木

### On-Call

on-call 不是榮譽勳章。  
如果一個系統的值班非常痛，通常不是人不夠扛，而是系統、流程、監控和變更管理設計不好。

### Go 實務映射

對 Go 服務團隊，這些常常具體體現在：

- 部署時是否看 error rate / latency canary
- 每個 alert 是否對應 runbook
- 是否能快速區分 app 問題、依賴問題、infra 問題
- 是否把臨時修補逐步收斂成簡化後的長期方案

## 7. 放到博彩與電商場景，R1 會變成什麼

### 場景一：電商大促

你會遇到：

- 商品頁流量暴增
- 下單路徑變慢
- cache miss 打爆 DB
- 某些熱門商品成為熱點

R1 會讓你問：

- 我們的核心 SLI 是商品頁延遲，還是下單成功率？
- 大促前有沒有做容量預估與壓測？
- alert 是不是會在真正影響訂單時才叫？
- 發佈是否避開活動高峰？

### 場景二：博彩熱門賽事

你會遇到：

- WebSocket 長連線很多
- 某些時刻盤口更新與下注同時暴增
- upstream feed 慢一點，整條鏈就堆住
- 使用者對延遲極度敏感

R1 會讓你問：

- 我們監控的是 CPU，還是下注成功率與盤口更新延遲？
- 容量規劃有沒有把長連線和瞬間尖峰一起算進去？
- incident 時先止血的策略是什麼？
- 值班的人有沒有足夠上下文快速判斷？

## 8. 這份筆記要怎麼接到 R4

可以把兩者理解成：

- `R1`: 告訴你什麼叫值得在意的問題
- `R4`: 告訴你這些問題在 Linux 與系統層是怎麼發生的

如果只看 R1，你可能會有管理視角但缺乏底層直覺。  
如果只看 R4，你可能很會講 `epoll`，但不一定知道為什麼團隊需要先定義 SLO、先做容量規劃、先設計好告警。

## 9. 面試版最短回答

### SLO、監控、容量規劃和高併發有什麼關係？

SLO 先定義什麼叫服務品質，監控負責在品質偏離時及早發現，容量規劃負責預估在流量上升時哪些資源會先撞牆，而高併發則是這些問題在線上最常出現的形式。它們不是分開的題目，而是一條完整的可靠性鏈。

### 為什麼 backend engineer 也要懂這些？

因為你寫的 timeout、重試、cache、SQL、部署方式、指標埋點，全部都會影響服務是否達成 SLO。真正成熟的 backend 工程師，不只寫功能，也會對可靠性結果負責。
